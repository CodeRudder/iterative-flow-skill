# 四轴交叉审查 (Four-Axis Cross-Review) — 详细指引

> 本文件是 SKILL.md Step 1B 的详细参考内容，在执行四轴交叉审查时按需加载。
> 设计背景见 [多视角交叉审查设计文档](../docs/multi-perspective-review-design.md)

---

## 四轴视角定义

| 轴 | Agent 视角 | 核心关注 | 审查其他轴时问什么 |
|----|-----------|---------|-----------------|
| Flow | 流程完整性 | 步骤、分支、交互、异常 | "Data 说状态 X 存在，但 Flow 哪步到达？" |
| Data | 数据一致性 | 实体、状态、字段、约束 | "Flow 步骤 3 扣了库存，但状态机有这行转换吗？" |
| Role | 职责清晰性 | 角色、权限、协作边界 | "Flow 这步没标注角色，谁执行？" |
| Rule | 条件完整性 | 业务规则、路由、校验 | "Flow 条件分支引用了哪条规则？有兜底吗？" |

## 执行流程（按 Phase 分批）

```
For each Phase (A, B, C, ...):

  Phase 1: 并行生成四轴输出
    Task(Flow):  生成当前 Phase 的 L2 子流程树
    Task(Data):  生成当前 Phase 涉及的实体状态变更
    Task(Role):  标注当前 Phase 的角色分配（按需）
    Task(Rule):  提取当前 Phase 的业务规则（按需）
    ↓ 最多 3 个并行，按需分批

  Phase 2: 交叉审查（单个 subagent 模拟四轴视角）
    Task(Cross-Reviewer): 读取四轴输出 → 依次模拟每个轴的视角审查其他轴 → 输出审查报告
    ↓

  Phase 3: 分类处理
    审查报告中的每个发现分为三类：
    A类（自动解决）: PRD 有明确答案 → 自动修复 + 记录决策 + PRD 出处
    R类（推荐方案）: 有行业标准/最佳实践 → 给推荐 + 理由 → 用户可一键采纳
    Q类（待用户决策）: PRD 未明确且无标准 → 整理问题列表 → 暂停等待用户
    ↓

  Phase 4: 输出决策日志
    写入 rounds/round-N/review/cross-review-{phase}.md
    格式见下方"决策日志格式"
    ↓

  Phase 5: 如有 Q 类问题 → 暂停 → 展示问题列表 → 等待用户回答 → 修复
    ↓

  Phase 6: 确认本 Phase 完成 → 进入下一个 Phase
```

## 交叉审查问题清单

审查 Agent 必须逐项检查以下清单：

**以 Flow 视角审查 Data：**
- [ ] Data 中每个状态，Flow 里都有步骤可达吗？（状态可达性）
- [ ] Data 的每条状态转换，Flow 里都有对应步骤触发吗？（转换触发）
- [ ] Data 的每个字段更新步骤，Flow 里存在吗？（字段覆盖）
- [ ] Data 的每种终止路径，Flow 都有对应分支吗？（终态收敛）

**以 Data 视角审查 Flow：**
- [ ] Flow 的"锁定/占用"操作，有对应的"释放"步骤吗？（操作配对）
- [ ] Flow 的每步操作，对实体产生了什么状态变更？（状态标注）
- [ ] 所有终止路径上，每种资源操作都配对了吗？（资源对称）
- [ ] Flow 引用的字段值，是从哪个步骤写入的？（字段来源）

**以 Rule 视角审查 Flow：**
- [ ] Flow 的每个条件分支，都引用了规则编号吗？（规则引用）
- [ ] 规则表的所有条件组合，Flow 都有对应路径吗？（覆盖完整）
- [ ] Flow 的条件分支最后有 else/默认路径吗？（兜底规则）
- [ ] 不同步骤引用的规则，有没有冲突？（规则矛盾）

**以 Role 视角审查 Flow：**
- [ ] 每个步骤都标注了执行角色吗？（角色标注）
- [ ] 前后步骤的角色切换合理吗？（角色连续性）
- [ ] 同一角色是否既审批又提交？（权限互斥）
- [ ] 每个角色至少负责一个步骤吗？（角色覆盖）

**跨轴交叉检查：**
- [ ] Flow × Data: 步骤操作与实体状态变更是否一一对应？
- [ ] Flow × Rule: 条件分支与规则表是否完全覆盖？
- [ ] Data × Rule: 规则触发的状态变更是否在状态机中有定义？
- [ ] Data × Role: 状态变更的操作者是否有权限？

## 决策日志格式

```markdown
## 决策日志 — Phase {X}: {阶段名称}

### 自动解决（{N} 项）
| # | 审查视角 | 发现的问题 | 解决方式 | 依据来源 | 影响步骤 |
|---|---------|-----------|---------|---------|---------|

### 推荐方案（{N} 项）
| # | 审查视角 | 发现的问题 | 推荐方案 | 推荐理由 | 状态(✓已采纳/?待确认) |
|---|---------|-----------|---------|---------|------|

### 待用户决策（{N} 项）
| # | 审查视角 | 问题描述 | 上下文 | 各视角意见 | 用户回答 |
|---|---------|---------|--------|-----------|---------|
```

## 审查产出文件

```
rounds/round-N/review/
├── cross-review-{phase-A}.md      # Phase A 的交叉审查报告 + 决策日志
├── cross-review-{phase-B}.md      # Phase B 的交叉审查报告 + 决策日志
├── ...
└── cross-review-summary.md        # 全局决策总览（所有 Phase 的 Q 类汇总）
```

## 与三代理评审 (Step 1A) 的关系

| 维度 | 三代理评审 (1A) | 四轴交叉审查 (1B) |
|------|----------------|-----------------|
| **追问者** | User/Dev/Integrator 三个角色 | Flow/Data/Role/Rule 四个轴 |
| **追问对象** | 流程文档本身 | 四轴输出之间的交叉间隙 |
| **关注点** | 用户体验 + 实施可行性 | 数据一致性 + 规则完整性 |
| **用户参与** | 无（纯AI审查） | Q类问题需用户决策 |
| **产出** | issues.md | 决策日志 + issues.md |
| **互补性** | 擅长发现交互和体验问题 | 擅长发现轴间脱节和遗漏 |

**方法 C（组合模式）中，先用 1B 发现结构间隙，再用 1A 发现体验和实施问题。**
