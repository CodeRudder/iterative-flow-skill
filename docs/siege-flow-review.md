# 攻城战完整流程步骤树 -- 开发可用性评估报告

> 评估对象：`examples/siege/攻城战-完整流程步骤树.md`
> 评估日期：2025-05-05

---

## 一、总体评价

文档整体结构清晰，主流程（4个Phase）到子流程的层级划分合理，树的深度控制在5级以内。交互行为（点击、弹窗、确认等）覆盖了主要用户路径。但经过逐树逐项检查，仍发现若干问题，按严重程度分类如下。

---

## 二、阻塞性问题（导致无法开发）

### B-1. 缺少 FL-SIEGE-02 攻城准备的完整子树

Tree 2（FL-SIEGE-02-01）只覆盖了"选择目标城池"，Tree 3（FL-SIEGE-02-02~04）覆盖了"创建编队"。但 Tree 1 主流程将 Phase A 标注为 `FL-SIEGE-02`，意味着 FL-SIEGE-02 应是一个完整的攻城准备子流程。文档中没有给出 FL-SIEGE-02 的顶层汇总树（类似 FL-SIEGE-03 那样的"部队出发→精灵移动→到达"三级结构）。开发者无法确认 FL-SIEGE-02 到底包含哪些子步骤、步骤间如何串联。

**影响**: Phase A 的入口和出口定义清晰，但内部子步骤的编号归属和层级关系不明确。Tree 2 的编号是 `FL-SIEGE-02-01`，Tree 3 的编号是 `FL-SIEGE-02-02~04`，暗示它们同属 FL-SIEGE-02，但文档没有显式声明这一关系，也没有给出 FL-SIEGE-02 的汇总树。

### B-2. 缺少 Tree 编号与 FL 编号的对照表

文档中 Tree 使用 `Tree 1~8` 的序号，同时每个 Tree 又有 `FL-SIEGE-XX` 的流程编号。两者映射关系如下但不明确：

| Tree 序号 | FL 编号 | 名称 |
|-----------|---------|------|
| Tree 1 | FL-SIEGE-01 | 攻城战主流程 |
| Tree 2 | FL-SIEGE-02-01 | 选择目标城池 |
| Tree 3 | FL-SIEGE-02-02~04 | 创建编队 |
| Tree 4 | FL-SIEGE-03 | 行军推进 |
| Tree 5 | FL-SIEGE-03-02 | 行军精灵移动 |
| Tree 6 | FL-SIEGE-03-02-03 | 路线交互 |
| Tree 7 | FL-SIEGE-04 | 攻城战斗 |
| Tree 7a | FL-SIEGE-04-02 | 攻击循环 |
| Tree 7b | FL-SIEGE-04-03 | 城池渲染阶段切换 |
| Tree 8 | FL-SIEGE-05 | 结算返回 |

问题：
- Tree 4 包含 `FL-SIEGE-03-01`、`FL-SIEGE-03-03` 两个子步骤，但 `FL-SIEGE-03-02` 又被拆成独立的 Tree 5，这种拆分方式导致编号不连续，开发者在实现 FL-SIEGE-03 时需要在 Tree 4 和 Tree 5 之间反复跳转。
- Tree 7a（FL-SIEGE-04-02）和 Tree 7b（FL-SIEGE-04-03）是 Tree 7（FL-SIEGE-04）的子步骤，但被提升为独立的 Tree，这种做法虽然控制了树深度，但打断了 FL-SIEGE-04 的完整性。

**影响**: 开发者需要自行拼接分散的 Tree 才能还原一个完整的子流程（如行军推进、攻城战斗），增加了理解和实现成本。

### B-3. Tree 5（行军精灵移动）回城后缺少与 Phase D 的衔接

Tree 5 的步骤 05（回城）描述了精灵回到出发城池、编队状态恢复空闲，流程结束。但 Tree 1 主流程 Phase B 明确标注"中途取消 -> Phase D 回城"，Phase D 的定义是"结算返回 -> Tree 8"。

问题是：Tree 5 的回城直接"流程结束"，不走 Tree 8 的结算流程。这与主流程的定义矛盾。取消行军后到底需不需要弹战报？需不需要解锁兵力？需不需要清除攻占任务？Tree 5 的步骤 05 没有提及清除攻占任务和行军实例，但 Tree 8 的步骤 05 有。如果取消行军不走 Tree 8，那么攻占任务和行军实例可能不会被正确清理。

**影响**: 这是一个关键的数据一致性漏洞。取消行军后的资源回收路径不明确，可能导致攻占任务泄漏、兵力永久锁定。

### B-4. 攻占任务的生命周期没有完整定义

攻占任务在 Tree 3 步骤 06 中创建，在 Tree 8 步骤 05 中清除。但在以下路径中，攻占任务的清理状态不明确：
- **取消行军**（Tree 5 步骤 05）：未提及是否清除攻占任务
- **城池已被己方占领**（Tree 4 步骤 03）：直接跳 Phase D 回城，但 Phase D 的回城是否走 Tree 8 的完整流程（包括清除攻占任务）？
- **跳过战斗**（Tree 7 玩家交互）：直接走到步骤 04 战斗结算，但"跳过"如何结算？按当前战况快照？还是直接判负？

**影响**: 攻占任务和行军实例的创建/销毁不对称，可能导致运行时状态泄漏。

## 三、重要问题（会导致实现偏差）

### I-1. Tree 4 到达检查与 Tree 5 到达处理存在重复

Tree 4（FL-SIEGE-03）步骤 03"到达目标城池"包含了以下操作：
1. 标记行军状态为"已到达"
2. 更新编队位置
3. 检查城池归属
4. 城池被己方占领 -> 提示 -> Phase D 回城

而 Tree 5（FL-SIEGE-03-02）步骤 04"到达处理"也包含：
1. 标记行军状态为"已到达"
2. 清除精灵和路线渲染
3. 直接触发攻城战斗

两处都执行了"标记行军状态为已到达"，但执行的操作不同。开发者不清楚应该先执行 Tree 5 的步骤 04 还是 Tree 4 的步骤 03。Tree 4 的步骤 03 里有城池归属检查，Tree 5 的步骤 04 里没有。实际执行顺序应该是：Tree 5-04（到达处理：精灵停止+清除渲染）-> Tree 4-03（归属检查），但文档没有说明这个执行顺序。

**影响**: 重复标记状态可能导致 Bug，且归属检查的时机不明确。

### I-2. Tree 5 步骤 04 直接跳转 FL-SIEGE-04，但 Tree 4 步骤 03 也有到达处理

Tree 5 步骤 04 写的是"直接触发 -> FL-SIEGE-04 攻城战斗"，但 Tree 4 步骤 03 写的是到达后先检查城池归属，可能跳转到 Phase D。两者形成冲突路径：Tree 5 说直接打，Tree 4 说先检查。开发者不知道该走哪条路。

**建议**: 明确 Tree 5 步骤 04 应调用 Tree 4 步骤 03 的逻辑（先检查归属再决定是否开战），而非直接跳转 FL-SIEGE-04。

### I-3. 城池信息面板缺少关闭按钮的描述

Tree 2 中，城池信息面板的关闭方式是"点击面板外区域"或"点击另一座城池"。但没有描述面板本身是否有关闭按钮（X）。如果面板较大遮挡了大部分地图，玩家可能无法方便地点击面板外区域。

**影响**: 交互设计不完整。

### I-4. 编队创建面板中"返回"按钮的异常流程不一致

Tree 3 的树形图中，异常分支写的是"玩家点击'返回' -> 弹出确认'放弃编队？' -> 确认 -> 关闭面板 -> Tree 2"。但下方的文字描述写的是"玩家点击'返回' -> 关闭面板 -> 返回地图"。一个说回 Tree 2（选择目标城池），一个说回地图，且一个有确认弹窗一个没有。

**影响**: 开发者无法确定返回行为的具体实现。

### I-5. 攻城战斗的终止条件存在歧义

Tree 7a 步骤 04 的终止判定：
- "守方兵力 <= 0 且 城墙HP <= 0" -> 攻方胜利
- "攻方兵力 <= 0" -> 攻方失败
- "战斗时间超时" -> 攻方失败

问题：
1. "守方兵力 <= 0 且 城墙HP <= 0"要求同时满足，但攻城战斗的逻辑是"城墙HP先扣减，城墙归零后伤害溢出到兵力"。这意味着城墙归零时守方兵力可能仍有大量剩余，需要继续攻击才能清零兵力。这与"城墙倒塌 -> 胜利"的直觉不符，且文档没有解释为什么需要双条件。
2. "战斗时间超时"的超时时长是多少？文档多次提到"参考 PRD"但没有给出具体值或默认值。
3. 缺少"玩家主动撤退"的终止条件。Tree 7 中有"跳过战斗"但没有"撤退"选项。"跳过"的含义不明确——是加速结算到最终结果，还是直接判负？

**影响**: 胜负判定逻辑实现可能偏离设计意图。

### I-6. 行军计时器在 Tree 4 和 Tree 5 中重复定义

Tree 4 步骤 01 写的是"启动行军计时器 -> Tree 5"，Tree 5 步骤 01 写的是精灵出场（不涉及计时器），步骤 02 写的是"启动行军移动定时器"。两个计时器是什么关系？是同一个还是两个不同的？如果是同一个，不应该在两处分别启动。

**影响**: 可能导致计时器重复创建或计时逻辑混乱。

### I-7. 数据流中兵力锁定/解锁不对称

- **锁定时机**（Tree 3 步骤 06）："锁定兵力（扣减可用库存）、扣减粮草"
- **解锁时机**（Tree 8 步骤 05）："解锁编队兵力（恢复可用库存）"

但解锁时恢复的是"原始库存"还是扣除伤亡后的"剩余库存"？文档写"恢复可用库存"，没有区分。如果编队战斗中有伤亡，解锁后库存应该只恢复剩余兵力，而非原始兵力。Tree 8 步骤 04 中有"更新攻方编队实际兵力（原兵力 - 伤亡）"，但步骤 05 中"解锁编队兵力"是否与步骤 04 的更新关联，没有明确说明。

另外，取消行军（Tree 5 步骤 05）时没有提及是否解锁兵力和退还粮草。

**影响**: 资源管理逻辑可能实现错误。

### I-8. 树形图中 Tree 4 的编号跳跃

Tree 4 的树形图中：
- 01 = FL-SIEGE-03-01（部队出发）
- 02 = FL-SIEGE-03-02（行军精灵移动） -> 引用 Tree 5
- 03 = FL-SIEGE-03-03（到达目标城池）

但 Tree 4 下方的详细描述中，只展开了 `FL-SIEGE-03-01` 和 `FL-SIEGE-03-03`，中间的 `FL-SIEGE-03-02` 被跳到 Tree 5 去了。树形图里 02 还有一句"玩家取消 -> Phase D 回城"，但这个取消操作的实际处理在 Tree 5 步骤 05 中，且走的是 Tree 5 自己的回城逻辑而非 Tree 8 的 Phase D。描述存在矛盾。

### I-9. 城池信息面板中"出发城"概念未提前定义

Tree 2 的城池信息面板展示"与出发城距离"，但此时玩家尚未进入编队创建流程，"出发城"如何确定？是默认取玩家当前所在的城池？还是需要玩家先选择出发城？文档没有说明出发城的确定逻辑。

**影响**: 开发者不知道出发城的数据来源。

## 四、建议改进（不影响开发但可以更清晰）

### S-1. 多处引用"参考 PRD"但没有链接

文档多次提到"具体计算方式参考 PRD"、"受距离、武将速度、天气、地形等因素影响"，但没有给出 PRD 文档的路径或链接。开发者在实现行军时间计算、伤害计算等核心公式时，需要额外寻找 PRD 文档。

### S-2. 攻击循环中的"~2秒"应明确为配置值

文档多处使用"~2秒"描述攻击间隔，这是一个约数。建议明确为可配置参数，如"攻击间隔（默认2秒，可通过配置表调整）"。

### S-3. 城池渲染阶段阈值建议用变量定义

Tree 7b 的 HP 阈值（100%、70%、40%、10%）是硬编码在流程中的。建议标注这些值来自配置表，方便后续调整。

### S-4. 编队状态机缺少状态转换图

编队有多个状态（空闲、出发中、行军中、攻城中、回城中），这些状态在不同 Tree 中被隐式修改，但没有一个统一的状态机定义或状态转换图。建议补充。

### S-5. 战报界面的"确认"按钮行为可以更明确

Tree 8 步骤 01 中"玩家点击'确认' -> 02"，但战报界面是否可以通过点击面板外区域关闭？如果玩家强制关闭（如返回键）会怎样？建议补充。

### S-6. 多支编队同时攻城的场景未提及

文档只描述了单支编队的攻城流程。如果玩家同时对同一城池派出多支编队，或者对不同城池同时攻城，流程是否适用？是否需要排队/并发控制？文档未涉及。

### S-7. Tree 6（路线交互）的精灵点击与城池点击可能冲突

在行军过程中，如果精灵刚好经过城池附近，玩家点击时是否可能误触城池而非精灵？文档未提及碰撞检测的优先级。

### S-8. 异常流程汇总表不完整

异常流程汇总表（文档末尾）缺少以下场景：
- Tree 3 中"玩家点击返回"的确认弹窗选"否"的情况
- Tree 7 中战斗时间超时的情况
- Tree 5 步骤 05 回城途中是否可以再次取消
- 编队创建过程中如果可用指挥官列表为空（已提到灰置按钮，但未列入汇总表）

### S-9. Tree 3 编队校验表中"指挥官无冲突"的描述过于模糊

校验项"该指挥官已在其他任务中"——是所有任务类型都冲突，还是只有攻城任务冲突？指挥官是否可以同时执行其他类型的任务（如采集、巡逻）？

### S-10. 行军路线"避开不可通行区域"的计算细节缺失

Tree 4 步骤 01 提到"计算行军路线（出发城 -> 目标城，避开不可通行区域）"，但没有描述寻路算法、不可通行区域的定义、路线是否一定存在等问题。

## 五、各 Tree 逐一审查结果

### Tree 1: 攻城战主流程
- **一致性**: Phase D 的描述（"结算返回 -> Tree 8"）与实际流程有出入。取消行军时 Tree 5 直接回城结束，不经过 Tree 8 的结算。
- **完整性**: 整体4个Phase覆盖了完整生命周期。

### Tree 2: 选择目标城池
- **交互闭环**: 城池信息面板的关闭路径完整（面板外点击、换城池）。确认弹窗的取消路径也有。
- **数据流**: "出发城"概念未定义。[I-9]
- **校验逻辑**: "不可攻打"的具体条件列出了3项，详细条件与文字描述一致。

### Tree 3: 创建编队
- **一致性**: 异常分支中"返回"行为不一致（回到 Tree 2 vs 回到地图）。[I-4]
- **数据流**: 攻占任务创建时的字段描述清晰。编队状态转换明确。
- **校验表**: 5项校验覆盖了主要合法性检查。
- **交互闭环**: "返回"的确认弹窗完整。校验失败的提示弹窗有关闭路径。

### Tree 4: 行军推进
- **一致性**: 与 Tree 5 存在到达处理的重复。[I-1, I-2]
- **完整性**: 部队出发、行军、到达三阶段完整。归属检查是亮点。
- **异常**: "城池已被己方占领"场景的回城路径不明确。[B-3]

### Tree 5: 行军精灵移动
- **一致性**: 回城后不接入 Phase D 的结算流程。[B-3]
- **数据流**: 行军实例状态转换完整。但回城时未提及攻占任务清理。[B-4]
- **交互闭环**: 精灵出场、移动、到达、回城四个子步骤的前置/后继关系正确。

### Tree 6: 路线交互
- **完整性**: 面板展示内容、关闭路径、取消确认流程都清晰。
- **交互闭环**: 面板外点击关闭、确认弹窗的确认/取消都有处理。精灵继续移动不受面板影响有说明。

### Tree 7: 攻城战斗
- **完整性**: 初始化、攻击循环、渲染切换、结束处理、玩家交互五个维度都有覆盖。
- **数据流**: 战报实例的生成字段清晰。编队实际兵力的更新有说明。
- **交互闭环**: 跳过战斗的确认弹窗完整。战斗信息面板和城池战况面板都有关闭路径。
- **歧义**: "跳过战斗"的结算逻辑不明确。[I-5]

### Tree 7a: 攻击循环
- **完整性**: 攻方攻击、守方反击、渲染检查、终止判定四步骤完整。
- **歧义**: 胜利条件"守方兵力 <= 0 且城墙HP <= 0"可能不符合设计意图。[I-5]
- **缺失**: 战斗时间超时的具体时长未给出。[I-5]

### Tree 7b: 城池渲染阶段切换
- **完整性**: 5个阶段的HP阈值和视觉效果描述详细。
- **执行步骤**: 渲染切换步骤清晰。
- **建议**: 阈值应标注为配置值。[S-3]

### Tree 8: 结算返回
- **数据流**: 战报展示 -> 奖励发放/城池归属 -> 回城 -> 编队恢复。完整覆盖。
- **交互闭环**: 战报界面有"确认"按钮。回城途中玩家可点击精灵查看信息。
- **问题**: 步骤 05 弹出提示"编队已返回"，但这个提示的关闭方式未说明。
- **资源回收**: 兵力解锁、攻占任务清除、行军实例清除都有提及。但兵力解锁与伤亡扣除的关系不明确。[I-7]

## 六、结论

### 是否可以直接用于开发？

**不完全可以。** 文档作为开发参考的整体质量不错，结构清晰、层级合理，但存在以下必须先解决的问题：

1. **必须修复的阻塞性问题**（4项）：
   - 取消行军后的完整清理路径（攻占任务、行军实例、兵力解锁、粮草退还）
   - Tree 4 与 Tree 5 的到达处理逻辑合并/明确顺序
   - FL-SIEGE-02 攻城准备子流程的顶层汇总树
   - 编号映射表的补充

2. **应当修复的重要问题**（9项）：
   - 特别是 I-1（到达处理重复）、I-2（冲突跳转路径）、I-4（返回行为不一致）、I-5（终止条件歧义）、I-7（兵力解锁逻辑）这几项

3. **建议改进**（10项）：
