# ERP 场景结构化流程树生成方法论

> 基于通用结构化流程树方法论 v2 的 ERP 领域适配版本
> 聚焦采购、审批、库存、财务等 ERP 核心流程的系统化建模
> 版本：v1.0 | 日期：2026-05-06

---

## 0. 核心设计原则

### 0.1 ERP 建模四轴：Flow + Data + Role + Rule

ERP 系统在通用"流程 + 数据"双轴基础上，新增**角色（Role）**和**规则（Rule）**维度。流程树描述系统"做了什么"，实体生命周期描述系统"操作什么"，角色轴描述"谁来做"，规则轴描述"什么条件下路由到哪里"。

| 轴 | 描述什么 | 输出物 |
|----|---------|--------|
| Flow 轴 | 步骤、分支、交互、异常 | 流程步骤树 |
| Data 轴 | 实体、状态、字段、约束 | 实体生命周期表 |
| Role 轴 | 执行角色、职责边界、协作关系 | 角色泳道图 |
| Rule 轴 | 审批路由、金额阈值、业务策略 | 业务规则决策表 |

### 0.2 四阶段生成流程

```
阶段一：需求分析 & 骨架提取
    |
阶段二：分层流程分解
    |
阶段三：穷举与补充
    |
阶段四：一致性验证
```

### 0.3 ERP 特有核心关注点

| 关注点 | 说明 | 基础方法论覆盖情况 |
|--------|------|-------------------|
| 多角色协作 | 同一流程涉及 3~5 个角色（申请人、主管、经理、财务、采购员等） | 需扩展 |
| 审批链路由 | 根据金额、部门、类别等条件动态决定审批路径 | 需扩展 |
| 单据链驱动 | PR -> PO -> GR -> Invoice -> Payment，每个单据触发下一个 | 实体生命周期可覆盖 |
| 业务规则表 | 将复杂的条件判断从流程树中提取为独立的决策表 | 需扩展 |
| 审计追溯 | 每个状态变更必须记录操作人、时间、操作、原因 | 需扩展 |
| 主数据校验 | 物料编码、供应商、成本中心、预算科目等需校验 | 需扩展 |
| 预算控制 | 预算占用/释放必须对称，跨申请单的预算竞争处理 | 资源对称性可覆盖 |

## 1. 阶段一：需求分析 & 骨架提取

**目标**：从 PRD 中提取 ERP 流程骨架、关键实体、参与角色和业务规则。

### 步骤 1.1：PRD 结构化阅读

从 PRD 中提取以下信息：

| 提取项 | 说明 | ERP 示例 |
|--------|------|---------|
| 功能边界 | 本流程的起点和终点 | "员工提交采购申请 -> 采购订单完成/作废" |
| 参与角色 | 参与流程的所有角色 | 申请人、直属主管、部门经理、财务、采购员 |
| 主要场景 | 正常路径（Happy Path） | 申请 -> 审批 -> 采购 -> 收货 -> 付款 |
| 关键约束 | 金额阈值、审批层级、预算限制 | "5000 元以上需部门经理审批" |
| 业务规则 | 路由规则、金额策略、权限矩阵 | "IT 类采购必须走 IT 部门审批" |
| 主数据依赖 | 物料、供应商、成本中心等 | "物料主数据、供应商名录、预算科目" |

### 步骤 1.2：主流程骨干提取

以 ERP 采购申请主流程为例，提取 3~7 个阶段：

```
主流程骨干（ERP 采购申请示例）
Phase A: 申请创建    -- 员工填写采购申请单，选择物料和数量
Phase B: 审批流转    -- 根据规则逐级审批，含驳回和修改循环
Phase C: 采购执行    -- 采购员创建采购订单，发送给供应商
Phase D: 收货入库    -- 仓库收货并入库，生成收货单
Phase E: 发票校验    -- 财务校验发票与收货单匹配
Phase F: 付款结算    -- 财务发起付款，完成采购闭环
```

**规则**（与基础方法论一致）：
- 每个阶段用一句话描述其核心目的
- 阶段按时间顺序排列
- 阶段数量控制在 3~7 个
- 每个阶段对应一棵 L2 子流程树

### 步骤 1.3：关键实体识别

ERP 场景下的核心实体及其生命周期：

| 实体 | 定义 | 初始状态 | 来源 |
|------|------|---------|------|
| 采购申请单 (PR) | 员工发起的采购需求凭证 | 草稿 | PRD 采购申请章节 |
| 审批记录 (Approval) | 一次审批动作的记录 | 待审批 | PRD 审批流程章节 |
| 采购订单 (PO) | 向供应商下达的正式采购合同 | 未创建 | PRD 采购订单章节 |
| 收货单 (GR) | 仓库收货确认的记录 | 未创建 | PRD 收货管理章节 |
| 发票 (Invoice) | 供应商开具的发票记录 | 未创建 | PRD 发票管理章节 |
| 付款单 (Payment) | 财务发起的付款指令 | 未创建 | PRD 付款结算章节 |
| 预算占用 (BudgetReservation) | 对预算的临时占用记录 | 未占用 | PRD 预算管理章节 |

### 步骤 1.4：角色识别与泳道划分

**ERP 方法论新增步骤**。明确识别所有参与角色及其职责边界：

| 角色 | 角色类型 | 职责范围 | PRD 来源 |
|------|---------|---------|---------|
| 申请人 (Requester) | 人工角色 | 填写申请、修改申请、撤回申请 | PRD 角色权限章节 |
| 直属主管 (Supervisor) | 人工角色 | 审批/驳回申请 | PRD 审批矩阵章节 |
| 部门经理 (DeptManager) | 人工角色 | 高金额审批、部门预算管控 | PRD 审批矩阵章节 |
| 采购员 (Buyer) | 人工角色 | 创建采购订单、选择供应商、跟踪交货 | PRD 采购执行章节 |
| 仓库管理员 (Warehouse) | 人工角色 | 收货、质检、入库 | PRD 收货管理章节 |
| 财务 (Finance) | 人工角色 | 发票校验、付款审批、预算管理 | PRD 财务结算章节 |
| 系统 (System) | 系统角色 | 自动校验、路由计算、状态流转、通知发送 | 系统设计 |

**角色标注规则**：
- 每个流程步骤必须标注执行角色
- 系统自动执行的步骤标注为 `[系统自动]`
- 角色标注格式：`步骤名称 [角色名]` 或 `步骤名称 [系统自动]`

### 步骤 1.5：业务规则提取

**ERP 方法论新增步骤**。将复杂的条件判断从流程描述中提取为独立规则：

| 规则编号 | 规则名称 | 条件 | 动作 | 优先级 |
|---------|---------|------|------|--------|
| R-01 | 低金额审批 | 金额 <= 5000 | 直属主管审批 | 1 |
| R-02 | 中金额审批 | 5000 < 金额 <= 50000 | 直属主管 -> 部门经理 | 2 |
| R-03 | 高金额审批 | 金额 > 50000 | 直属主管 -> 部门经理 -> 总经理 | 3 |
| R-04 | IT 类特殊审批 | 类别 = IT | 额外经过 IT 部门审批 | 4 |
| R-05 | 预算超额拦截 | 申请金额 > 剩余预算 | 拦截并提示预算不足 | 5 |

**规则提取原则**：
- 将金额阈值、部门策略等条件判断从流程树中分离
- 规则表独立维护，流程树引用规则编号
- 规则变更只需修改规则表，不影响流程结构

### 步骤 1.6：建立需求可追溯性矩阵（草稿）

与基础方法论一致，为每个 PRD 需求分配唯一 ID：

| PRD ID | 需求描述 | 对应阶段 | 流程节点（待填充） |
|--------|---------|---------|------------------|
| REQ-PR-01 | 员工可提交采购申请 | Phase A | -- |
| REQ-PR-02 | 申请金额决定审批层级 | Phase B | -- |
| REQ-PR-03 | 审批通过后自动生成采购订单 | Phase C | -- |
| REQ-PO-01 | 采购员可选择供应商 | Phase C | -- |
| REQ-GR-01 | 收货数量与采购订单匹配校验 | Phase D | -- |
| REQ-PAY-01 | 三单匹配（PO+GR+Invoice）后付款 | Phase F | -- |

## 2. 阶段二：分层流程分解

**目标**：将骨干逐层分解为可执行的流程步骤树，重点处理 ERP 特有的审批路由和单据链驱动。

### 步骤 2.1：生成 L1 主流程

以 ERP 采购流程为例：

```
FL-PR-01 ERP 采购主流程
|-- Phase A: 申请创建 -> FL-PR-02
|-- Phase B: 审批流转 -> FL-PR-03
|   |-- 异常: 驳回 -> 申请人修改 -> FL-PR-02（修改后重新提交）
|   |-- 异常: 撤回 -> 申请关闭 -> FL-PR-CANCEL
|-- Phase C: 采购执行 -> FL-PR-04
|-- Phase D: 收货入库 -> FL-PR-05
|-- Phase E: 发票校验 -> FL-PR-06
|-- Phase F: 付款结算 -> FL-PR-07
|-- FL-PR-CANCEL: 申请取消（异常路径）
```

**L1 主流程组成**（与基础方法论一致）：元信息、流程概览、阶段详情。

### 步骤 2.2：生成 L2 子流程

以 Phase B 审批流转为例：

```
FL-PR-03 审批流转子流程
|-- 01 系统计算审批路径 [系统自动]
|   |-- 读取申请金额、类别、部门
|   |-- 匹配业务规则表（R-01~R-05）
|   |-- 生成审批人列表（有序）
|-- 02 直属主管审批 [直属主管]
|   |-- 查看待审批申请列表
|   |-- 查看申请详情（物料、数量、金额、用途）
|   |-- 审批决策：通过 / 驳回（附原因）
|-- 03 部门经理审批（条件：金额 > 5000） [部门经理]
|   |-- 查看待审批申请
|   |-- 校验部门预算余额
|   |-- 审批决策：通过 / 驳回（附原因）
|-- 04 总经理审批（条件：金额 > 50000） [总经理]
|   |-- 查看待审批申请
|   |-- 审批决策：通过 / 驳回（附原因）
|-- 05 审批通过后处理 [系统自动]
|   |-- 更新申请单状态为"已审批"
|   |-- 占用预算额度
|   |-- 触发采购订单创建（Phase C）
|   |-- 记录审计日志
|-- 异常: 驳回 -> 通知申请人 -> 修改后重新提交（回到 01）
|-- 异常: 审批超时（72 小时） -> 自动提醒审批人
```

### 步骤 2.3：审批链建模规则

**ERP 方法论核心规则**。ERP 系统的审批链是复杂流程的核心，必须按以下模式建模：

#### 审批链路由模式

每个审批环节必须明确路由规则：

```
审批链建模规范：

模式一：顺序审批（Sequential）
  审批人A -> 审批人B -> 审批人C
  适用：每个审批人必须独立审批

模式二：条件路由（Conditional Routing）
  if 金额 <= 5000 then 直属主管
  if 5000 < 金额 <= 50000 then 直属主管 -> 部门经理
  if 金额 > 50000 then 直属主管 -> 部门经理 -> 总经理
  适用：根据业务属性动态决定审批路径

模式三：会签（Counter-sign）
  所有会签人必须同意才通过
  适用：跨部门采购需多部门共同审批
```

#### 审批结果处理

每个审批步骤必须同时包含通过和驳回两条路径：

```
审批结果处理规范：
1. 审批通过 -> 流转到下一审批人（或触发后续动作）
2. 审批驳回 -> 附带驳回原因，通知申请人
   - 驳回到申请人：申请人可修改后重新提交
   - 驳回到上一步：修改后重新进入当前审批
3. 驳回原因必须记录在审批记录中（审计要求）
```

**注**：审批转办和紧急模式属于高级功能，此处仅提及不在流程树中详细建模。

### 步骤 2.4：单据链建模规则

**ERP 方法论核心规则**。ERP 的核心特征是单据链驱动，一个单据的完成触发下一个单据的创建。

```
单据链驱动模式：

PR（采购申请）.status = 已审批
  -> 触发创建 PO（采购订单）

PO（采购订单）.status = 已下达
  -> 触发供应商确认流程

PO（采购订单）.status = 已确认
  -> 触发 GR（收货单）创建准备

GR（收货单）.status = 已收货
  -> 触发 Invoice（发票）校验准备

Invoice（发票）.status = 已校验
  -> 触发 Payment（付款单）创建

Payment（付款单）.status = 已完成
  -> 采购流程闭环
```

**单据链建模规则**：
- 每个单据是一个独立的实体，拥有自己的完整生命周期
- 单据之间的触发关系用 `DocumentA.status = X -> trigger DocumentB 创建` 表达
- 单据链中的每个环节可以建模为独立的子流程
- 单据之间通过单据编号关联（PR No. -> PO No. -> GR No.）

### 步骤 2.5：子树拆分规则

与基础方法论一致，补充 ERP 特有拆分时机：

| 规则 | 说明 |
|------|------|
| 最大深度 | 每棵树不超过 5 级 |
| 审批链拆分 | 审批流转逻辑独立拆为子树 |
| 单据链拆分 | 每个单据的生命周期独立拆为子树 |
| 规则引用拆分 | 复杂规则校验独立拆为子树 |
| 节点数量 | 每层 3~9 个节点 |

### 步骤详情的必填字段

与基础方法论一致，ERP 场景增加以下必填/可选字段：

| 字段 | 适用场景 | ERP 示例 |
|------|---------|---------|
| 执行角色 | 所有步骤 | "直属主管"、"[系统自动]" |
| 规则引用 | 涉及条件路由的步骤 | "依据规则 R-01~R-03" |
| 主数据校验 | 涉及主数据操作的步骤 | "校验物料编码是否有效" |
| 预算操作 | 涉及金额变更的步骤 | "占用预算科目 B-2024-001" |
| 审计记录 | 涉及状态变更的步骤 | "记录审批通过日志" |

## 3. 阶段三：穷举与补充

**目标**：对阶段二产出的流程树进行穷举补充，重点覆盖 ERP 特有的审批路径、预算控制和审计追溯。

### 步骤 3.1：交互行为穷举

对流程树中的每个步骤，逐一检查以下交互类型：

| 交互类型 | 检查项 | ERP 示例 |
|---------|--------|---------|
| 提交/发送 | 用户提交了什么表单？ | "提交采购申请单" |
| 审批操作 | 审批人执行了什么操作？ | "通过/驳回/转办" |
| 确认弹窗 | 确认/取消两条路径是否都有？ | "提交确认弹窗：确认提交 / 继续编辑" |
| 页面跳转 | 从哪跳到哪？触发条件？ | "审批通过后跳转采购订单创建页" |
| 选择/切换 | 选择后数据如何更新？ | "选择供应商后自动填充联系方式和付款条件" |
| 表单输入 | 输入校验规则？失败提示？ | "数量必须为正整数，失败提示请输入有效数量" |

**弹窗闭环检查**（与基础方法论一致）：每个弹窗必须有明确的弹出条件、关闭方式和背景流程说明。

### 步骤 3.2：异常路径穷举

对每个正常步骤，系统性检查异常维度：

| 异常维度 | 检查问题 | ERP 示例 |
|---------|---------|---------|
| 驳回 | 审批驳回后如何处理？ | "驳回后通知申请人，附驳回原因，可修改重提" |
| 校验失败 | 主数据校验失败时如何处理？ | "物料编码无效 -> 提示选择有效物料 -> 停留在当前页" |
| 超时 | 审批超时如何处理？ | "72 小时未审批 -> 自动提醒审批人" |
| 状态冲突 | 单据状态冲突时怎么办？ | "申请单已审批，不允许修改" |
| 预算不足 | 预算不够时如何处理？ | "剩余预算不足 -> 拦截并提示 -> 无法提交" |
| 主数据缺失 | 物料/供应商不存在怎么办？ | "物料未维护 -> 提示先维护物料主数据" |

**异常汇总表**（文档末尾必须有）：

```
| 触发步骤 | 异常条件 | 处理方式 | 恢复步骤 |
```

### 步骤 3.3：采购申请单状态机建模

**ERP 方法论核心步骤**。采购申请单是 ERP 采购流程的核心单据，必须作为一等实体建模。

#### 采购申请单状态机定义

```
采购申请单状态机（PR State Machine）

状态列表：
  S1: 草稿 (draft)             -- 申请单正在编辑中
  S2: 待审批 (pending_approval) -- 已提交，等待审批
  S3: 审批中 (in_approval)      -- 正在审批链中流转
  S4: 已审批 (approved)         -- 审批通过，等待转采购订单
  S5: 已转单 (converted)        -- 已生成采购订单
  S6: 已驳回 (rejected)         -- 审批被驳回
  S7: 已撤回 (withdrawn)        -- 申请人主动撤回
  S8: 已取消 (cancelled)        -- 申请单已取消

状态转换表：
  S1 -> S2:  申请人提交           -- FL-PR-02-05
  S2 -> S3:  系统分配审批人       -- FL-PR-03-01
  S3 -> S3:  当前审批人通过       -- FL-PR-03-02~04（流转到下一审批人）
  S3 -> S4:  最终审批人通过       -- FL-PR-03-05
  S3 -> S6:  任一审批人驳回       -- FL-PR-03-REJECT
  S6 -> S1:  申请人修改后重新编辑 -- FL-PR-02-EDIT
  S2 -> S7:  申请人撤回          -- FL-PR-CANCEL
  S4 -> S5:  生成采购订单        -- FL-PR-04-01
  S7 -> S8:  撤回完成关闭        -- FL-PR-CANCEL
```

**状态转换完整性检查（ESA 方法）**：

对采购申请单执行 Events x States 笛卡尔积检查：

```
对于实体 PR:
  对于每个事件 Ev（提交、通过、驳回、撤回、转单、取消）:
    对于每个状态 St (S1~S8):
      Ev 在 St 下是否有效？
      如果有效 -> 转到哪个状态？在哪个流程步骤中定义？
      如果无效 -> 是否需要防护？是否需要提示？
```

#### 关键规则：
- 每个单据的**创建**步骤必须明确（PR 在 FL-PR-02-01 创建）
- 每个单据的**终态**步骤必须明确（converted、cancelled 为终态）
- **所有终止路径**都必须到达终态，不可停留在中间态
- **禁止非法跳转**：如从 draft 直接跳到 approved 必须经过审批流程

### 步骤 3.4：其他核心单据生命周期

#### 采购订单状态机

```
采购订单（PO）
  状态: draft -> pending -> confirmed -> partially_received -> received -> closed
  draft -> pending:         采购员提交采购订单
  pending -> confirmed:     供应商确认订单
  confirmed -> partially_received:  部分收货
  confirmed -> received:    全部收货
  partially_received -> received:  收货完成
  received -> closed:       发票校验和付款完成
```

#### 收货单状态机

```
收货单（GR）
  状态: not_created -> pending -> quality_check -> accepted -> rejected
  not_created -> pending:       仓库接收送货通知
  pending -> quality_check:     开始质检
  quality_check -> accepted:    质检通过，入库
  quality_check -> rejected:    质检不合格，退货
```

#### 付款单状态机

```
付款单（Payment）
  状态: not_created -> pending_review -> approved -> paid -> failed
  not_created -> pending_review:  财务创建付款单
  pending_review -> approved:     付款审批通过
  approved -> paid:               银行转账完成
  approved -> failed:             付款失败（银行返回错误）
```

### 步骤 3.5：预算操作对称性检查

**ERP 方法论核心检查项**。预算操作必须严格对称：

```
预算对称性检查表：

| 操作类型       | 占用步骤           | 正常释放（扣减）      | 取消释放             | 异常释放             |
|--------------|------------------|-------------------|--------------------|--------------------|
| 预算占用       | FL-PR-03-05      | FL-PR-04-01 (转PO扣减) | FL-PR-CANCEL (释放)  | 驳回 (释放)          |
| 预算扣减       | FL-PR-04-01      | -- (终态)           | PO取消 (释放)        | --                  |
| 预算释放       | FL-PR-CANCEL     | -- (终态)           | --                  | --                  |
```

**预算对称规则**：
- `已占用 = 已扣减 + 已释放`（数学恒等式，在任意时刻必须成立）
- 审批通过时占用预算，驳回/撤回时释放预算
- 转采购订单时将占用转为扣减
- 采购订单取消时释放已扣减的预算
- 跨申请单的预算竞争：当多个申请同时占用同一预算科目时，需要处理并发占用问题

### 步骤 3.6：审计追溯建模

**ERP 方法论核心要求**。审计追溯不是独立的流程步骤，而是贯穿所有状态变更的横切关注点。

#### 审计日志规范

```
审计日志格式：

每条状态变更必须记录：
  - 时间戳（timestamp）: 操作发生时间
  - 操作人（operator）:  执行操作的人员/系统
  - 操作（action）:      提交/审批通过/驳回/撤回/取消等
  - 原因（reason）:      操作原因（驳回时必填）
  - 前状态（from_status）: 变更前状态
  - 后状态（to_status）:   变更后状态
  - 单据编号（doc_no）:    关联的单据编号
```

#### 审计记录时机

| 触发场景 | 必须记录的内容 | 示例 |
|---------|-------------|------|
| 单据状态变更 | 全部字段 | PR-2024-001 由 draft 变为 pending_approval |
| 审批操作 | 操作人、结果、原因 | 张三(直属主管) 通过审批 |
| 预算操作 | 操作类型、金额、科目 | 占用预算科目 B-001 金额 5000 |
| 主数据校验 | 校验项、结果 | 物料编码 MAT-001 校验通过 |

## 4. 阶段四：一致性验证

**目标**：系统性验证流程树的完整性、一致性和可开发性，加入 ERP 特有验证维度。

### 步骤 4.1：需求覆盖验证（RTM）

更新阶段一创建的追溯矩阵，确保每个 PRD 需求都有对应的流程节点：

```
验证条件：
- 每个 PRD 需求 ID 至少有一个对应的流程节点
- 每个流程节点都有 PRD 来源
- PRD 中的边界条件已体现为异常分支
- ERP 特有需求（审批规则、预算控制、单据链）全部覆盖
```

### 步骤 4.2：交互闭环验证

| 检查项 | 验证方法 |
|--------|---------|
| 确认弹窗闭环 | 每个确认弹窗都有确认和取消两条路径 |
| 审批结果闭环 | 每个审批步骤都有通过和驳回两条路径 |
| 表单提交闭环 | 提交成功和校验失败都有处理 |
| 页面跳转闭环 | 每个跳转都有明确的触发条件和目标 |
| 无死胡同 | 每个分支都有明确的出口 |

### 步骤 4.3：数据生命周期验证

| 检查项 | 验证方法 |
|--------|---------|
| 单据创建 | 每种单据都有明确的创建步骤 |
| 单据终态 | 每种单据都能到达合法的终态 |
| 全路径清理 | 所有终止路径（正常/驳回/取消）都清理了单据和预算 |
| 预算对称 | 预算占用/扣减/释放配对完整 |

### 步骤 4.4：ERP 特有验证

**以下为 ERP 方法论新增验证项**，必须逐一检查：

#### 4.4.1 角色覆盖验证

```
检查项：
- 每个流程步骤都标注了执行角色
- 每个角色至少负责一个步骤
- 系统自动步骤标注为 [系统自动]
- 人工角色步骤不包含系统自动逻辑
- 角色权限边界清晰（申请人不能审批自己的单据）
```

#### 4.4.2 业务规则完整性验证

```
检查项：
- 所有条件路由都有对应的规则编号
- 规则表覆盖所有条件组合（金额区间、类别、部门）
- 规则之间无矛盾
- 规则优先级无歧义
- 默认规则（兜底规则）是否存在
```

#### 4.4.3 审批路径枚举验证

```
检查项：
- 所有金额阈值的审批路径都已枚举
- 每个审批路径都有明确的起点和终点
- 驳回后的处理路径完整（回到哪一步？可修改什么？）
- 审批超时的处理规则明确
- 会签场景的通过/驳回逻辑完整
```

#### 4.4.4 预算对称性验证

```
检查项：
- 已占用 = 已扣减 + 已释放（数学恒等式）
- 所有驳回路径是否都释放了预算
- 所有撤回/取消路径是否都释放了预算
- 采购订单取消路径是否释放已扣减预算
- 是否存在预算占用后无法释放的死路径
```

#### 4.4.5 审计追溯完整性验证

```
检查项：
- 所有状态变更步骤是否都标注了审计记录
- 审计日志格式是否符合规范
- 驳回操作是否记录了驳回原因
- 预算操作是否记录了金额和科目
- 主数据校验是否记录了校验结果
```

#### 4.4.6 单据链完整性验证

```
检查项：
- 单据链中的每个环节都有触发条件
- 前一单据的完成状态与后一单据的创建触发条件一致
- 单据链中的关联字段（PR No. -> PO No.）传递完整
- 单据链中间环节取消时，上下游单据状态如何联动
```

### 步骤 4.5：跨树引用一致性

| 检查项 | 验证方法 |
|--------|---------|
| 前向引用 | A 树说"-> FL-PR-04"，B 树确实是 FL-PR-04 的定义 |
| 后向引用 | B 树的"前置"与 A 树的"后继"匹配 |
| 状态一致 | A 树设置的单据状态与 B 树期望的前置状态一致 |
| 无悬空引用 | 所有引用的 FL 编号都有对应定义 |
| 无矛盾路径 | A 树说"直接结束"，主流程说"应经过 B 树" -- 禁止 |
| 规则引用一致 | 流程步骤引用的规则编号在规则表中都有定义 |

### 步骤 4.6：步骤详情质量检查

| 检查项 | 要求 |
|--------|------|
| 每个步骤有目的 | 一句话说明作用 |
| 关键步骤格式 | 编号列表，每行一个动作，动词开头 |
| 角色标注完整 | 每个步骤都标注了执行角色 |
| 不含 UI 细节 | 不描述颜色、像素、布局 |
| 不含不确定数值 | 用"参考 PRD"替代不确定的公式和数值 |
| 前置/后继明确 | 每个步骤都知道从哪来、到哪去 |

### 6.2 ERP 场景 ToT 分支穷举

对 ERP 流程中的决策点，使用 Tree of Thought 展开：

```
在步骤 FL-PR-03-01（审批路径计算）中，有以下决策点：
  决策点 1: 申请金额范围
    |-- 金额 <= 5000 -> 直属主管审批 -> FL-PR-03-02
    |-- 5000 < 金额 <= 50000 -> 直属主管 -> 部门经理 -> FL-PR-03-02~03
    |-- 金额 > 50000 -> 直属主管 -> 部门经理 -> 总经理 -> FL-PR-03-02~04
  决策点 2: 申请类别
    |-- 普通采购 -> 按金额路由
    |-- IT 类采购 -> 额外经过 IT 部门审批（规则 R-04）
    |-- 固定资产 -> 额外经过资产管理部门
```

## 7. 质量检查清单（ERP 版）

### 结构检查
- [ ] 每棵树深度不超过 5 级
- [ ] 超深的子树已拆分
- [ ] 编号体系统一，无跳跃或冲突
- [ ] 每棵树都有树形图 + 步骤详情
- [ ] 每个步骤标注了执行角色

### 需求追溯
- [ ] 所有 PRD 需求都有对应的流程节点
- [ ] 所有流程节点都能追溯到 PRD 来源
- [ ] 追溯矩阵已填写完整
- [ ] 审批规则、金额阈值、预算控制等 ERP 特有需求全部覆盖

### 角色检查
- [ ] 所有流程步骤都标注了执行角色
- [ ] 系统自动步骤标注为 [系统自动]
- [ ] 角色权限边界清晰（不能越权操作）
- [ ] 申请人不能审批自己的单据
- [ ] 每个角色至少负责一个步骤

### 规则检查
- [ ] 所有条件路由都引用了规则编号
- [ ] 规则表覆盖所有条件组合
- [ ] 规则之间无矛盾
- [ ] 存在默认兜底规则
- [ ] 规则优先级无歧义

### 数据检查
- [ ] 采购申请单状态机完整定义，覆盖所有状态和事件
- [ ] 采购订单、收货单、付款单等单据状态机完整
- [ ] 状态转换覆盖所有终止路径
- [ ] 预算占用/扣减/释放对称
- [ ] 单据字段更新步骤明确

### 审计检查
- [ ] 所有状态变更有审计日志记录
- [ ] 审计日志格式符合规范
- [ ] 驳回操作记录了驳回原因
- [ ] 预算操作记录了金额和科目
- [ ] 主数据校验记录了校验结果

### 异常检查
- [ ] 每个步骤的异常分支都已列出
- [ ] 异常汇总表覆盖了所有异常
- [ ] 驳回/撤回路径的数据清理完整（含预算释放）
- [ ] 审批超时等约束有明确值或标注"参考 PRD"
- [ ] 主数据缺失等异常有处理方案

### 单据链检查
- [ ] 单据链中每个环节都有触发条件
- [ ] 前单据完成状态与后单据创建触发条件一致
- [ ] 单据关联字段传递完整（PR No. -> PO No. -> GR No.）
- [ ] 单据链中间环节取消时上下游联动正确

### 跨树一致性
- [ ] 前向引用与定义匹配
- [ ] 后向引用与前置匹配
- [ ] 规则引用与规则表匹配
- [ ] 无悬空引用
- [ ] 无矛盾路径

### 步骤详情
- [ ] 每个步骤有一句话目的
- [ ] 关键步骤是编号列表，每行一个动作
- [ ] 每个步骤标注了执行角色
- [ ] 不包含 UI 设计细节
- [ ] 不包含不确定数值
- [ ] 涉及预算的步骤有预算操作说明
- [ ] 涉及主数据的步骤有校验说明
