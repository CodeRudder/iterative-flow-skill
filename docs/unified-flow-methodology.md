# 统一结构化流程树生成方法论 v1

> 综合 游戏/电商/ERP 三大领域适配版本的最佳实践，形成领域无关的通用方法论
> 基于 BPMN 2.0、UML State Machine、Event Storming、IREB Layered FSM 等研究成果
> 版本：v1.0 | 日期：2026-05-06

---

## 0. 核心设计原则

### 0.1 统一四轴模型：Flow + Data + Role + Rule

软件需求可沿四个正交维度完整描述。不同领域对各轴的依赖程度不同，但四轴模型本身是通用的。

| 轴 | 描述什么 | 输出物 | 来源 |
|----|---------|--------|------|
| Flow 轴 | 步骤、分支、交互、异常 | 流程步骤树 | 基础方法论 |
| Data 轴 | 实体、状态、字段、约束 | 实体生命周期表 | 基础方法论 |
| Role 轴 | 执行角色、职责边界、协作关系 | 角色泳道图 | ERP 扩展 |
| Rule 轴 | 业务规则、路由策略、校验规则 | 业务规则决策表 | ERP 扩展 |

**原则**：Flow 描述"做了什么"，Data 描述"操作什么"，Role 描述"谁来做"，Rule 描述"什么条件下路由到哪里"。四轴独立建模、交叉验证。

### 0.2 四阶段生成流程

```
阶段一：需求分析 & 骨架提取
    |                    阶段二：分层流程分解
    |                        |                    阶段三：穷举与补充
    |                        |                        |                    阶段四：一致性验证
```

### 0.3 领域适配指南表

| 领域 | Flow轴 | Data轴 | Role轴 | Rule轴 | 外部集成 | 审计追溯 |
|------|--------|--------|--------|--------|---------|---------|
| 游戏/娱乐 | 高 | 高 | 低(单人) | 中 | 低 | 低 |
| 电商/交易 | 高 | 高 | 中 | 高 | 高(支付/物流) | 中 |
| ERP/企业 | 高 | 高 | 高(多角色) | 高(复杂规则) | 中 | 高 |
| 通用SaaS | 高 | 高 | 中 | 中 | 中 | 中 |

**使用方式**：根据目标领域，将本方法论中标注"(按需)"的步骤按此表决定是否执行。

### 0.4 方法论来源映射

| 本方法论要素 | 来源 |
|-------------|------|
| L1→L2→L3 分层分解 | BPMN 2.0 Hierarchical Decomposition |
| 实体状态机建模 | UML State Machine / IREB Layered FSM |
| 需求可追溯矩阵 | IEEE 29148 |
| 事件驱动识别 | Event Storming |
| Events x States 完整性检查 | IREB ESA (Event-State Analysis) |
| 角色泳道划分 | BPMN Swimlane / UML Activity Diagram |
| 业务规则决策表 | Decision Table (DMN) |
| 四阶段 Prompt 策略 | CoT + ToT + Few-Shot |

## 1. 阶段一：需求分析 & 骨架提取

**目标**：从 PRD 中提取流程骨架、关键实体、参与角色和业务规则，建立可追溯性框架。

### 1.1 PRD 结构化阅读

从 PRD 中提取以下信息：

| 提取项 | 说明 | 示例 |
|--------|------|------|
| 功能边界 | 本流程的起点和终点 | "用户提交订单 → 订单完成/关闭" |
| 用户角色 | 参与流程的角色 | 玩家、买家、申请人、审批人 |
| 主要场景 | 正常路径（Happy Path） | 选商品 → 下单 → 支付 → 收货 |
| 关键约束 | 时间、资源、金额等约束 | "订单 30 分钟未支付自动取消" |
| 业务规则 | 路由、校验、计算规则 | "5000 元以上需部门经理审批" |
| 外部依赖 | 依赖的外部系统或数据 | 支付网关、物流 API、物料主数据 |

### 1.2 主流程骨干提取

借鉴 User Story Mapping 的 Backbone 概念，提取 3~7 个**阶段（Phase）**作为主流程骨架。每个阶段用一句话描述核心目的，按时间顺序排列，每个阶段对应一棵 L2 子流程树。

### 1.3 关键实体识别

识别流程中涉及的所有**数据实体**，为每个实体定义初始状态：

| 实体 | 定义 | 初始状态 | 来源 |
|------|------|---------|------|
| {实体名} | {一句话定义} | {初始状态} | PRD {章节} |

### 1.4 角色识别与泳道划分（按需）

**来源：ERP 扩展**。多角色流程必须执行此步骤。

| 角色 | 角色类型 | 职责范围 | PRD 来源 |
|------|---------|---------|---------|
| {角色名} | 人工角色/系统角色 | {职责描述} | PRD {章节} |

**角色标注规则**：每个流程步骤标注 `[角色名]` 或 `[系统自动]`。参考领域适配指南表，Role轴为"高"的领域必须执行。

### 1.5 业务规则提取（按需）

**来源：ERP 扩展**。规则驱动的流程必须执行此步骤。

| 规则编号 | 规则名称 | 条件 | 动作 | 优先级 |
|---------|---------|------|------|--------|
| R-01 | {名称} | {条件表达式} | {执行动作} | {优先级} |

**规则提取原则**：将条件判断从流程树中分离为独立规则表；规则变更只修改规则表，不影响流程结构；必须存在默认兜底规则。Rule轴为"高"的领域必须执行。

### 1.6 外部集成点识别（按需）

**来源：电商扩展**。涉及外部系统交互的流程必须执行此步骤。

| 外部系统 | 交互类型 | 我方职责 | 对方职责（不建模） |
|---------|---------|---------|-----------------|
| {系统名} | 同步调用/异步回调 | {我方做什么} | {不描述内部逻辑} |

**集成边界规则**：只描述本系统调用和回调；不描述外部系统内部逻辑；使用"调用 → 等待回调"模式。

### 1.7 需求可追溯性矩阵（草稿）

为每个 PRD 需求分配唯一 ID，建立到流程阶段的初始映射：

| PRD ID | 需求描述 | 对应阶段 | 流程节点（待填充） |
|--------|---------|---------|------------------|
| REQ-{模块}-{序号} | {需求描述} | Phase {字母} | -- |

## 2. 阶段二：分层流程分解

**目标**：将骨干逐层分解为可执行的流程步骤树。

### 2.1 生成 L1 主流程

L1 主流程是全局路线图，只包含阶段和阶段间的关系。由元信息（版本、触发/结束条件、前置/后置状态）、流程概览（ASCII 树形图）和阶段详情（每个 Phase：目的 + 关键步骤 + 前置/后继 + 异常）组成。

```
FL-{模块}-01 {功能名称}主流程
|-- Phase A: {阶段名} -> FL-{模块}-02
|-- Phase B: {阶段名} -> FL-{模块}-03
|-- Phase C: {阶段名} -> FL-{模块}-04
|   |-- 异常: {条件} -> Phase D
```

### 2.2 生成 L2 子流程

每个 Phase 展开为独立的 L2 子流程树，由元信息、流程导航、步骤详情和异常汇总组成。

```
FL-{模块}-{序号} {子流程名称}
|-- 01 步骤名称 [角色名]
|   |-- 子操作 A
|   |-- -> FL-{模块}-{序号}-{子序号}（引用 L3）
|-- 02 步骤名称 [角色名]
|-- 异常: 条件 -> 处理 -> 转至
```

### 2.3 生成 L3 详细步骤

对 L2 中交互密集或逻辑复杂的步骤，进一步展开为 L3。

### 2.4 子树拆分规则

| 规则 | 说明 |
|------|------|
| 最大深度 | 每棵树不超过 **5 级** |
| 拆分时机 | 当某节点的子步骤需要独立树时，拆为子树 |
| 引用标记 | 父树中用 `-> FL-{编号}` 标注 |
| 节点数量 | 每层 3~9 个节点 |
| 异步拆分（按需） | 异步回调处理独立拆为子树 |
| 审批链拆分（按需） | 审批流转逻辑独立拆为子树 |
| 单据链拆分（按需） | 每个单据生命周期独立拆为子树 |

### 2.5 角色标注规则（按需）

**来源：ERP 扩展**。每个步骤标注 `[角色名]` 或 `[系统自动]`；角色权限边界清晰（互斥规则）；每个角色至少负责一个步骤。

### 2.6 业务规则引用规则（按需）

**来源：ERP 扩展**。条件路由步骤引用规则编号（`依据规则 R-01~R-03`）；不在流程树中硬编码条件判断。

### 2.7 外部系统边界规则（按需）

**来源：电商扩展**。使用"调用 → 等待回调"模式描述异步交互；只描述我方调用和回调。

### 2.8 异步流程两阶段建模（按需）

**来源：电商扩展**。每个异步操作建模为两步：

**步骤 A（发起请求）**：构造参数 → 调用外部接口 → 记录请求 ID → 设置超时定时器 → 状态变更为 pending。

**步骤 B（接收回调）**：接收回调 → 验证签名 → 幂等检查 → 若终态则跳过 → 执行业务逻辑 → 清除定时器 → 返回确认。

### 步骤详情的必填字段

| 字段 | 要求 | 示例 |
|------|------|------|
| 一句话目的 | 说明本步骤的作用 | "计算审批路径并分配审批人" |
| 关键步骤 | 编号列表，每行动词开头 | "1. 读取申请金额和类别" |
| 来源 | 关联 PRD 章节 | "[PRD 3.2.2](link)" |
| 前置/后继 | 从哪来、到哪去 | "<- FL-XX-YY → FL-XX-ZZ" |
| 异常分支 | 条件 → 处理 → 转至 | "超时 → 查询结果 → FL-TIMEOUT" |

可选字段（按需）：执行角色、规则引用、幂等键、超时时间、补偿操作、预算/库存操作、审计记录、主数据校验。

## 3. 阶段三：穷举与补充

**目标**：对流程树进行穷举补充，确保不遗漏步骤、交互、异常和数据操作。

### 3.1 交互行为穷举

对每个步骤检查：点击/提交、弹窗/面板、确认弹窗（确认+取消两条路径）、页面跳转、选择/切换、表单输入、审批操作（按需）。

**弹窗闭环检查**：每个弹窗必须有弹出条件、至少一种关闭方式、背景流程影响说明。

### 3.2 异常路径穷举

| 异常维度 | 检查问题 |
|---------|---------|
| 用户取消 | 能否取消？取消后如何清理？ |
| 校验失败 | 提示什么？停留在哪？ |
| 超时 | 超时后如何处理？ |
| 状态冲突 | 前置条件不满足时怎么办？ |
| 资源不足 | 如何提示和处理？ |
| 外部系统异常（按需） | 外部不可用时如何降级？ |
| 驳回（按需） | 驳回后如何修改重提？ |

**异常汇总表**（文档末尾必须有）：触发步骤 | 异常条件 | 处理方式 | 恢复步骤

### 3.3 实体生命周期建模

**核心步骤，所有领域必须执行**。为每个实体定义：状态列表（含初始态和终态）、状态转换表（含触发条件和步骤引用）、关键字段（含更新步骤）。

#### 状态转换完整性检查（ESA 方法）

对每个实体执行 **Events x States 笛卡尔积**检查：对每个事件在每个状态下是否有效？有效则转到哪个状态？无效则是否需要防护？

**关键要求**：每个实体有明确的创建步骤和终态；所有终止路径都到达终态；禁止非法跳转。

### 3.4 资源对称性检查

对每种资源操作确认配对完整：

| 操作类型 | 锁定/占用步骤 | 正常释放 | 取消释放 | 异常释放 |
|---------|-------------|---------|---------|---------|
| 资源 A | FL-XX-01 | FL-XX-05 | FL-XX-CANCEL | -- |

**检查规则**：所有终止路径都有释放；数学恒等式 `已锁定 = 已扣减 + 已释放` 恒成立；退还量说明明确；并发竞争已考虑。适用：游戏（行动力）、电商（库存）、ERP（预算）等。

### 3.5 审批链/决策链穷举（按需）

**来源：ERP 扩展**。三种模式：顺序审批、条件路由、会签。每个审批步骤必须包含通过和驳回两条路径，驳回必须记录原因。

### 3.6 审计追溯建模（按需）

**来源：ERP 扩展**。审计追溯是横切关注点。每条状态变更记录：时间戳、操作人、操作、原因（驳回必填）、前/后状态、单据编号。记录时机：状态变更、审批操作、预算/库存操作、主数据校验。

### 3.7 幂等性规则（按需）

**来源：电商扩展**。异步回调必须处理幂等：每个请求携带唯一幂等键；接收回调先查状态；终态直接返回成功不重复处理；所有回调记录日志。

## 4. 阶段四：一致性验证

**目标**：系统性验证流程树的完整性、一致性和可开发性。

### 4.1 需求覆盖验证

每个 PRD 需求 ID 至少有一个对应流程节点；每个流程节点有 PRD 来源；边界条件已体现为异常分支。

### 4.2 交互闭环验证

| 检查项 | 验证方法 |
|--------|---------|
| 弹窗闭环 | 每个弹窗都有至少一种关闭方式 |
| 确认弹窗 | 确认和取消两条路径都有 |
| 页面跳转 | 触发条件和目标明确 |
| 审批结果闭环（按需） | 通过和驳回两条路径 |
| 外部跳转闭环（按需） | 有回跳或超时处理 |
| 无死胡同 | 每个分支有明确出口 |

### 4.3 数据生命周期验证

每种实体有创建步骤和终态；所有终止路径都清理了实体；资源锁定/解锁、占用/释放配对完整；单据链完整（按需）。

### 4.4 资源对称性验证

`已锁定 = 已扣减 + 已释放` 恒成立；所有取消/超时路径都释放资源；退款路径退还已扣减资源；无死路径；并发场景已处理。

### 4.5 跨树引用一致性

前向引用与定义匹配；后向引用与前置匹配；状态一致（A 树设置的状态 = B 树期望的前置状态）；无悬空引用；无矛盾路径；异步配对一致（按需）；规则引用一致（按需）。

### 4.6 角色覆盖验证（按需）

**来源：ERP 扩展**。每个步骤标注了执行角色；每个角色至少负责一个步骤；系统步骤标注 `[系统自动]`；权限边界清晰；互斥规则已执行。

### 4.7 规则完整性验证（按需）

**来源：ERP 扩展**。所有条件路由有规则编号；规则表覆盖所有条件组合；规则无矛盾；优先级无歧义；存在默认兜底规则。

### 4.8 步骤详情质量检查

每个步骤有一句话目的；关键步骤为编号列表且动词开头；不含 UI 细节和不确定数值；前置/后继明确；角色标注完整（按需）；异步步骤成对（按需）。

## 5. 文档规范

### 5.1 文档产出清单

```
{功能名称}-结构化流程/
|-- L1-{功能名称}-主流程.md          -- L1 主流程
|-- L2-{子流程名称}.md               -- L2 子流程（每个 Phase 一个）
|-- L3-{详细步骤名称}.md             -- L3 详细步骤（按需）
|-- 实体生命周期.md                  -- 实体状态机
|-- 业务规则表.md                    -- 决策表（按需）
|-- 角色泳道说明.md                  -- 角色权限（按需）
|-- 集成边界说明.md                  -- 外部交互（按需）
|-- 需求追溯矩阵.md                  -- RTM
|-- 异常流程汇总.md                  -- 异常路径汇总
```

### 5.2 流程树格式

```
根节点名称（FL-XX 流程名）
|-- 01 步骤名称 [执行角色]
|   |-- 子操作 A
|   |-- -> 子树引用（FL-XX-YY）
|-- 02 步骤名称 [执行角色]
|-- 异常: 条件 -> 处理 -> 转至
```

分支规则：条件分支用 `->`；异常以 `异常:` 开头放最后；并行用 `（与 XX 并行）`；多角色流程每步标 `[角色名]`；异步用"发起 → 等待回调"。

### 5.3 实体生命周期格式

```markdown
## 实体：{名称}
> 定义 / PRD 来源

### 状态列表
| 状态 | 说明 | 控制变量值 |

### 状态转换表
| 当前状态 | 事件/条件 | 目标状态 | 执行角色 | 触发步骤 | 伴随操作 |

### 关键字段
| 字段 | 类型 | 初始值 | 更新步骤 |

### 审计日志记录点（按需）
| 步骤 | 记录内容 | 操作角色 |
```

### 5.4 业务规则表格式（按需）

```markdown
## 业务规则表：{名称}
> 适用范围 / PRD 来源

### 规则定义
| 规则编号 | 条件 | 动作 | 优先级 | 备注 |

### 默认规则
| 条件 | 动作 |
| 以上均不匹配 | 兜底处理 |
```

### 5.5 集成边界说明格式（按需）

```markdown
## 集成边界：{外部系统名}
> 我方角色 / 接口文档

### 我方发起的调用
| 调用场景 | 接口名称 | 请求参数 | 预期响应 | 调用步骤 |

| 回调场景 | 回调参数 | 幂等键 | 处理步骤 |

### 超时与补偿
| 场景 | 超时时间 | 补偿方式 | 补偿步骤 |
```

### 5.6 需求追溯矩阵格式

| PRD ID | 需求描述 | 流程节点 | 覆盖状态 |
|--------|---------|---------|---------|
| REQ-XX-01 | {描述} | FL-XX-01, FL-XX-02 | 已覆盖 |
| REQ-XX-02 | {描述} | -- | 未覆盖 |

## 6. Prompt 策略

### 6.1 统一四阶段 Prompt 模板

```
=== 阶段一：需求分析 ===
请阅读以下 PRD 片段，输出：
1. 功能边界（起点和终点）
2. 主流程骨干（3-7 个 Phase，每个一句话）
3. 关键实体列表（名称、定义、初始状态）
4. 参与角色列表【多角色流程必填】
5. 业务规则提取【规则驱动流程必填】
6. 外部系统集成点【有外部集成必填】
7. 需求 ID 列表

=== 阶段二：分层分解 ===
基于阶段一输出：
1. 生成 L1 主流程（ASCII 树 + 阶段详情）
2. 对每个 Phase 生成 L2 子流程
3. 对需要展开的步骤生成 L3

规则：树深度不超过 5 级；每个步骤有目的、关键步骤、前置/后继、异常分支；
不写 UI 细节和不确定数值；多角色标注 [角色名]；规则引用编号不硬编码；
异步建模为"发起 + 回调"；资源标注操作类型。

=== 阶段三：穷举补充 ===
3.1 交互穷举：点击/弹窗/跳转是否遗漏？
3.2 异常穷举：取消/超时/校验失败是否遗漏？
3.3 实体生命周期：每个实体状态转换表，所有终止路径到达终态。
3.4 资源对称：锁定/扣减/释放配对对称。
3.5 审批路径枚举【按需】
3.6 审计追溯【按需】
3.7 幂等性检查【按需】

=== 阶段四：一致性验证 ===
4.1 需求覆盖：所有 PRD 需求有对应流程节点？
4.2 交互闭环：所有弹窗有关闭方式？
4.3 数据生命周期：所有实体有完整状态机和终态？
4.4 资源对称：已锁定 = 已扣减 + 已释放？
4.5 跨树引用：所有 FL 编号有对应定义？
4.6 角色覆盖【按需】：所有步骤标注角色？
4.7 规则完整性【按需】：所有路由引用规则编号？
4.8 步骤质量：每个步骤有目的和前置/后继？
```

### 6.2 ToT 分支穷举技巧

对决策点使用 Tree of Thought 展开所有分支，每个分支继续展开后续步骤直到到达已定义的终止点。

### 6.3 自验证循环

生成完成后，让 AI 扮演 QA 工程师逐项检查：步骤前置/后继完整？弹窗有关闭方式？实体到达终态？资源配对释放？分支无矛盾？角色标注完整（多角色）？规则引用完整（规则驱动）？异步配对（外部集成）？幂等处理（电商）？审计记录（ERP）？输出问题列表。

## 7. 质量检查清单（统一版）

### 结构检查
- [ ] 每棵树深度不超过 5 级，超深子树已拆分
- [ ] 编号体系统一，无跳跃或冲突
- [ ] 每棵树都有树形图 + 步骤详情

### 需求追溯
- [ ] 所有 PRD 需求有对应流程节点，所有节点可追溯 PRD
- [ ] 追溯矩阵已填写完整

### 交互检查
- [ ] 所有可交互元素已列出，弹窗有关闭方式
- [ ] 确认弹窗有确认和取消两条路径，页面跳转有说明

### 数据检查
- [ ] 每种实体有完整状态转换表，覆盖所有终止路径
- [ ] 资源锁定/解锁对称，字段更新步骤明确

### 异常检查
- [ ] 每个步骤异常分支已列出，汇总表覆盖所有异常
- [ ] 取消/中断路径数据清理完整，约束有明确值或"参考 PRD"

### 跨树一致性
- [ ] 前向/后向引用匹配，无悬空引用，无矛盾路径

### 角色检查（按需）
- [ ] 所有步骤标注执行角色，权限边界清晰，互斥规则已执行

### 规则检查（按需）
- [ ] 所有条件路由引用规则编号，规则覆盖完整无矛盾，有兜底规则

### 审计检查（按需）
- [ ] 所有状态变更有日志，驳回有原因，资源操作有详情

### 外部集成检查（按需）
- [ ] 异步操作拆为"发起 + 回调"，回调有幂等处理，有超时补偿

### 步骤详情
- [ ] 每步有一句话目的，关键步骤为编号列表，不含 UI 细节和不确定数值

## 8. 领域适配速查表

根据目标领域快速确定必选/可选步骤。

### 游戏/娱乐领域

| 步骤 | 必选/可选 | 说明 |
|------|---------|------|
| 1.4 角色识别与泳道划分 | 可选 | 通常单人角色 |
| 1.5 业务规则提取 | 可选 | 规则相对简单 |
| 1.6 外部集成点识别 | 可选 | 外部集成较少 |
| 2.5~2.8 分解扩展规则 | 可选 | 视场景简化 |
| 3.5 审批链穷举 | 跳过 | 不适用 |
| 3.6 审计追溯 / 3.7 幂等性 | 可选 | 需求低 |
| 4.6~4.7 验证扩展 | 可选 | 视场景简化 |

### 电商/交易领域

| 步骤 | 必选/可选 | 说明 |
|------|---------|------|
| 1.4 角色识别与泳道划分 | 可选 | 角色较少 |
| 1.5 业务规则提取 | **必选** | 促销/优惠/限购规则复杂 |
| 1.6 外部集成点识别 | **必选** | 支付/物流是核心 |
| 2.6 规则引用规则 | **必选** | 促销规则独立维护 |
| 2.7 外部系统边界 / 2.8 异步建模 | **必选** | 支付回调、物流推送 |
| 3.5 审批链穷举 | 跳过 | 通常不适用 |
| 3.7 幂等性规则 | **必选** | 支付/退款回调必须幂等 |
| 4.7 规则完整性验证 | **必选** | 确保规则覆盖完整 |

### ERP/企业领域

| 步骤 | 必选/可选 | 说明 |
|------|---------|------|
| 1.4 角色识别与泳道划分 | **必选** | 3~5 角色协作是核心 |
| 1.5 业务规则提取 | **必选** | 审批路由、金额阈值 |
| 2.5 角色标注规则 | **必选** | 每步标注执行角色 |
| 2.6 规则引用规则 | **必选** | 条件路由引用规则表 |
| 3.5 审批链穷举 | **必选** | 审批链是核心 |
| 3.6 审计追溯建模 | **必选** | 企业合规要求 |
| 4.6 角色覆盖验证 | **必选** | 角色标注完整 |
| 4.7 规则完整性验证 | **必选** | 规则表覆盖完整 |
| 1.6 外部集成 / 2.8 异步 / 3.7 幂等 | 可选 | 视场景而定 |

### 通用 SaaS 领域

根据业务特征灵活组合以上三个领域的速查表。一般建议：角色识别和外部集成视场景而定，业务规则和审计追溯为中等优先级。
