# 流程步骤树生成规则

> 从 `examples/siege/攻城战-完整流程步骤树.md` 案例提炼，用于指导 AI 生成同等质量的结构化流程文档。

---

## 1. 结构规则

### 1.1 树深度与拆分
- 每棵流程树最大深度 **5级**，超过时拆分为独立子树
- 拆分时机：当某个节点的子步骤本身需要展开为独立树时（如攻击循环、渲染切换），将其提取为子树并在父树中用引用替代
- 子树命名：父树编号 + 字母后缀，如 Tree 7 → Tree 7a、Tree 7b

### 1.2 层级定义
| 层级 | 含义 | 示例 |
|------|------|------|
| Level 1 | 流程/子流程根节点 | FL-SIEGE-04 攻城战斗 |
| Level 2 | 主要阶段/步骤 | 01 攻城初始化、02 攻击循环 |
| Level 3 | 子步骤/操作 | 精灵播放攻击动画、计算伤害 |
| Level 4 | 操作细节/分支 | 扣减城墙HP、伤害数字飘出 |
| Level 5 | 最细粒度（分支结果、渲染细节） | 更新HP条、触发技能特效 |

### 1.3 编号体系
- 主流程：`FL-{模块}-{序号}`（如 FL-SIEGE-01）
- 子流程：`FL-{模块}-{序号}-{子序号}`（如 FL-SIEGE-03-02）
- 子步骤：在树形图中用 `01/02/03...` 简写，在步骤详情中用完整编号

### 1.4 每棵树的组成
每棵树必须包含：
1. **树形图**：ASCII 树形结构，展示所有节点和分支
2. **步骤详情**：每个 Level 2 节点展开为独立小节
3. **异常分支表**：条件 → 处理 → 转至

## 2. 内容规则

### 2.1 每个步骤节点必须包含
| 字段 | 要求 | 示例 |
|------|------|------|
| 一句话目的 | 说明本步骤的作用，不描述怎么做 | "驱动精灵沿路线持续移动并检测终点" |
| 关键步骤 | 编号列表，每个步骤一行 | "1. 启动行军移动定时器..." |
| 前置 | 从哪个步骤进入，附条件 | "← FL-SIEGE-03-01（行军实例已创建）" |
| 后继 | 跳转到哪个步骤 | "→ FL-SIEGE-03-02-04" |

### 2.2 可选字段（按需添加）
| 字段 | 适用场景 | 示例 |
|------|---------|------|
| ⏱ 时间 | 步骤有耗时/间隔/超时约束 | "每~2秒触发一次攻击" |
| 📱 空间 | 步骤涉及视口/场景/布局变化 | "攻城在天下地图上进行，玩家可自由拖拽" |
| 异常分支表 | 步骤有条件分支 | "进度到达1.0 → 停止定时器 → 04" |

### 2.3 关键步骤的编写要求
- **每个动作一行**：不要合并多个操作到一行
- **动词开头**：计算、扣减、更新、播放、弹出、启动、停止...
- **数据流向清晰**：明确什么数据从哪里来、到哪里去
- **不写具体数值**：用"参考 PRD"替代不确定的计算公式
- **不写 UI 设计**：不描述颜色、像素、布局等视觉细节

## 3. 交互穷举规则

### 3.1 所有交互行为必须显式列出
流程树中不能省略任何中间交互环节，包括但不限于：

| 交互类型 | 必须列出 | 示例 |
|---------|---------|------|
| 点击 | 玩家点击任何可交互元素 | "玩家点击'攻城'按钮" |
| 弹窗/面板 | 所有弹出和关闭行为 | "弹出城池信息面板" / "玩家点击面板外区域 → 关闭面板" |
| 确认弹窗 | 弹出、确认、取消三个路径 | "弹出确认弹窗 → 确认 → ... / 取消 → ..." |
| 跳转/切换 | 页面/场景/面板的切换 | "关闭编队面板 → 跳转天下地图" |
| 选择/切换 | 选择、切换、取消选择 | "玩家点击另一指挥官 → 切换选中" |
| 拖拽/滚动 | 拖动、缩放等操作 | "玩家拖拽浏览天下地图" |
| 展开/收起 | 折叠面板的展开和收起 | "展开兵力配置区域" / "收起配置区域" |

### 3.2 弹窗/面板的闭环检查
每个弹窗/面板必须满足：
- 有明确的**弹出条件**（什么操作触发）
- 有明确的**关闭方式**（至少一种）：
  - 点击关闭按钮
  - 点击面板外区域
  - 点击确认/取消
  - 系统自动关闭（说明条件）
- 面板打开期间，背景流程是否受影响必须说明

### 3.3 确认弹窗的标准模式
```
弹出确认弹窗"XXX？"
├── 玩家点击"确认" → 执行操作 → 后续步骤
└── 玩家点击"取消" → 关闭弹窗 → 停留/返回
```

## 4. 数据生命周期规则

### 4.1 每种数据实体必须定义完整生命周期
文档中涉及的每种实体（攻占任务、行军实例、编队等）必须明确：

| 阶段 | 必须说明 |
|------|---------|
| 创建 | 在哪个步骤创建，初始字段有哪些 |
| 读取 | 哪些步骤读取了该实体的数据 |
| 更新 | 哪些步骤修改了实体的状态/字段 |
| 销毁 | 在哪个步骤清除，是否有前置条件 |

### 4.2 状态转换必须一致
- 同一个实体的状态流转在所有引用处必须一致
- 编队状态示例：空闲 → 出发中 → 行军中 → 攻城中 → 回城中 → 空闲
- 每次状态变更必须在关键步骤中显式写出

### 4.3 资源操作必须对称
- **锁定**资源的步骤必须对应一个**解锁**步骤
- **扣减**资源的步骤，在异常退出时必须有**退还**路径
- 检查清单：
  - 兵力锁定 → 哪里解锁？（正常完成 / 取消 / 失败）
  - 粮草扣减 → 取消时是否退还？
  - 攻占任务创建 → 哪里清除？（所有终止路径都要覆盖）

### 4.4 跨树的引用必须一致
- A 树说"→ FL-SIEGE-XX"，B 树必须是 FL-SIEGE-XX 的定义
- A 树说"← Phase D"，Phase D 的入口条件必须匹配
- **禁止**：A 树的步骤说"直接结束"，而主流程定义该路径应经过 B 树

## 5. 异常流程规则

### 5.1 异常分支的三要素
每个异常分支必须包含：
1. **触发条件**：什么情况下发生
2. **处理方式**：系统做什么（提示/回退/重试/终止）
3. **恢复步骤**：流程跳转到哪里

### 5.2 异常汇总表
文档末尾必须有全局异常流程汇总表，覆盖所有异常：
- 每行格式：`触发步骤 | 异常条件 | 处理方式 | 恢复步骤`
- 检查每个树形图中的异常分支是否都被汇总表收录
- 检查每个弹窗/确认的"取消"路径是否被收录

### 5.3 取消/中断路径的完整性
从流程的任何中间状态，玩家都可能取消/退出。需要检查：
- 取消后的数据清理是否完整（任务、实例、资源锁定）
- 取消后的 UI 状态是否恢复（面板关闭、精灵清除）
- 取消后跳转到的目标是否合理

## 6. 树形图绘制规则

### 6.1 格式规范
```
根节点名称
├── 01 步骤名称
│   ├── 子步骤/操作
│   ├── 子步骤/操作
│   └── 子步骤/操作
├── 02 步骤名称
│   └── ...
└── 异常: 条件 → 处理
```

### 6.2 分支表示
- 条件分支用 `→` 连接结果和跳转目标
- 异常分支放在最后，以 `异常:` 开头
- 并行行为用括号标注，如 `（与02并行）`
- 子树引用用 `→ Tree X` 或 `→ Tree Xa` 标注

### 6.3 步骤命名
- 使用"动词+名词"格式：精灵出场、沿路线移动、到达处理
- 避免使用抽象名词：不写"处理逻辑"，写"到达处理"
- 树形图中可加括号补充说明：`03 路线交互（点击查看进度 / 取消）`

## 7. 质量检查清单

生成流程文档后，逐项检查：

### 结构检查
- [ ] 每棵树深度 ≤ 5 级
- [ ] 超深的子树已拆分
- [ ] 编号体系统一，无跳跃或冲突
- [ ] 每棵树都有树形图 + 步骤详情

### 交互检查
- [ ] 所有用户可点击的元素都有列出
- [ ] 所有弹窗/面板都有关闭方式
- [ ] 所有确认弹窗都有确认和取消两条路径
- [ ] 场景/面板切换都有说明

### 数据检查
- [ ] 每种实体都有完整的创建→读取→更新→销毁生命周期
- [ ] 资源锁定/解锁对称
- [ ] 所有终止路径（正常/取消/异常）都清理了资源
- [ ] 跨树引用一致，无矛盾

### 异常检查
- [ ] 每个步骤的异常分支都已列出
- [ ] 异常汇总表覆盖了所有异常
- [ ] 取消/中断路径的数据清理完整
- [ ] 超时/上限等约束条件有明确值或标注"参考 PRD"

### 步骤详情检查
- [ ] 每个步骤都有一句话目的
- [ ] 关键步骤是编号列表，每个动作一行
- [ ] 前置/后继关系正确
- [ ] 不包含 UI 设计细节（颜色、像素、布局）
- [ ] 不包含不确定的数值（用"参考 PRD"替代）
