# 结构化流程树生成方法论 v2

> 从 PRD 到开发可用流程文档的系统化方法
> 基于 BPMN 2.0、UML State Machine、Event Storming、IREB Layered FSM 等研究成果重新设计
> 版本：v2.0 | 日期：2026-05-06

---

## 0. 核心设计原则

### 0.1 双轴建模：Flow + Data

软件需求 = **流程（Flow）** + **数据（Data）**。流程树描述系统"做了什么"，实体生命周期描述系统"操作什么"。两者独立建模、交叉验证。

| 轴 | 描述什么 | 输出物 |
|----|---------|--------|
| Flow 轴 | 步骤、分支、交互、异常 | 流程步骤树 |
| Data 轴 | 实体、状态、字段、约束 | 实体生命周期表 |

### 0.2 四阶段生成流程

```
阶段一：需求分析 & 骨架提取
    ↓
阶段二：分层流程分解
    ↓
阶段三：穷举与补充
    ↓
阶段四：一致性验证
```

**关键改进**（相比 v1 规则体系）：
- 从"规则列表"升级为"有序流程"——AI 可逐步执行
- 实体生命周期建模成为一等公民——解决资源泄漏问题
- 需求可追溯性矩阵成为强制产物——确保 PRD 覆盖
- 验证阶段嵌入流程——而非事后补查

### 0.3 方法论来源映射

| 本方法论要素 | 来源 |
|-------------|------|
| L1→L2→L3 分层分解 | BPMN 2.0 Hierarchical Decomposition |
| 实体状态机建模 | UML State Machine / IREB Layered FSM |
| 需求可追溯矩阵 | IEEE 29148 |
| 事件驱动识别 | Event Storming |
| 交互穷举模式 | UI Flow Screen Flow |
| Events × States 完整性检查 | IREB ESA (Event-State Analysis) |
| 5-9 活动/层 经验规则 | BPMN Best Practices |
| 4阶段 Prompt 策略 | CoT + ToT + Few-Shot |

## 1. 阶段一：需求分析 & 骨架提取

**目标**：从 PRD 中提取流程骨架和关键实体，建立可追溯性框架。

### 步骤 1.1：PRD 结构化阅读

从 PRD 中提取以下信息：

| 提取项 | 说明 | 示例 |
|--------|------|------|
| 功能边界 | 本流程的起点和终点 | "玩家选择副本 → 结算完成" |
| 用户角色 | 参与流程的角色 | 玩家、系统 AI |
| 主要场景 | 正常路径（Happy Path） | 选副本 → 编队 → 战斗 → 结算 |
| 关键约束 | 时间、空间、资源约束 | "战斗每回合约 2 秒"、"副本中不能退出" |
| 外部依赖 | 依赖的其他系统或数据 | "英雄库"、"装备系统" |

### 步骤 1.2：主流程骨干提取

借鉴 User Story Mapping 的 Backbone 概念，提取 3~7 个**阶段（Phase）**作为主流程骨架：

```
主流程骨干（示例）
Phase A: 副本选择    — 玩家选择目标副本和难度
Phase B: 队伍编成    — 玩家组建出战队伍
Phase C: 副本战斗    — 回合制/即时战斗过程
Phase D: 结算奖励    — 发放奖励，更新记录
```

**规则**：
- 每个阶段用**一句话**描述其核心目的
- 阶段按时间顺序排列
- 阶段数量控制在 3~7 个（BPMN 最佳实践）
- 每个阶段对应一棵 L2 子流程树

### 步骤 1.3：关键实体识别

识别流程中涉及的所有**数据实体**，为每个实体定义初始状态：

| 实体 | 定义 | 初始状态 | 来源 |
|------|------|---------|------|
| 副本实例 | 一次副本挑战的运行时实例 | 不存在 | PRD §副本系统 |
| 队伍 | 玩家编成的出战阵容 | 空闲 | PRD §队伍系统 |
| 奖励包 | 副本通关后待领取的奖励 | 未生成 | PRD §奖励系统 |
| 行动力 | 进入副本消耗的资源 | 可用 | PRD §资源系统 |

### 步骤 1.4：建立需求可追溯性矩阵（草稿）

为每个 PRD 需求分配唯一 ID，建立到流程阶段的初始映射：

| PRD ID | 需求描述 | 对应阶段 | 流程节点（待填充） |
|--------|---------|---------|------------------|
| REQ-D-01 | 玩家可选择不同难度的副本 | Phase A | — |
| REQ-D-02 | 队伍至少包含 1 名英雄 | Phase B | — |
| REQ-D-03 | 战斗以回合制进行 | Phase C | — |
| REQ-D-04 | 通关后发放奖励 | Phase D | — |

## 2. 阶段二：分层流程分解

**目标**：将骨干逐层分解为可执行的流程步骤树。

### 步骤 2.1：生成 L1 主流程

L1 主流程是全局路线图，只包含阶段和阶段间的关系：

```
FL-{模块}-01 {功能名称}主流程
├── Phase A: {阶段名} → FL-{模块}-02
├── Phase B: {阶段名} → FL-{模块}-03
├── Phase C: {阶段名} → FL-{模块}-04
│   └── 异常: {条件} → Phase D
└── Phase D: {阶段名} → FL-{模块}-05
```

**L1 主流程的组成**：

| 部分 | 内容 |
|------|------|
| 元信息 | 版本、触发条件、结束条件、前置/后置状态 |
| 流程概览 | ASCII 树形图（仅含 Phase 和跳转） |
| 阶段详情 | 每个 Phase 一段：目的 + 关键步骤 + 前置/后继 + 异常 |

### 步骤 2.2：生成 L2 子流程

每个 Phase 展开为一棵独立的 L2 子流程树：

```
FL-{模块}-{序号} {子流程名称}
├── 01 步骤名称
│   ├── 子操作 A
│   ├── 子操作 B
│   └── → FL-{模块}-{序号}-{子序号}（引用 L3）
├── 02 步骤名称
│   └── ...
└── 异常: 条件 → 处理 → 转至
```

**L2 子流程的组成**：

| 部分 | 内容 |
|------|------|
| 元信息 | 归属功能、触发条件、结束条件、前置/后置状态、来源 PRD 章节 |
| 流程导航 | 上级引用 + 本级树形图 |
| 步骤详情 | 每个 Level-2 步骤展开为独立小节 |
| 异常汇总 | 触发步骤 / 异常条件 / 处理方式 / 恢复步骤 |

### 步骤 2.3：生成 L3 详细步骤

对于 L2 中需要进一步展开的步骤（通常是交互密集或逻辑复杂的步骤），生成 L3：

```
FL-{模块}-{序号}-{子序号} {详细步骤名称}
├── 01 子步骤
├── 02 子步骤
│   ├── 操作细节
│   └── 条件分支
└── 异常: 条件 → 处理
```

### 步骤 2.4：子树拆分规则

| 规则 | 说明 |
|------|------|
| 最大深度 | 每棵树不超过 **5 级** |
| 拆分时机 | 当某节点的子步骤本身需要独立树时，拆为子树 |
| 子树命名 | 父树编号 + 字母后缀（Tree 7 → Tree 7a） |
| 引用标记 | 父树中用 `→ FL-{编号}` 或 `→ Tree {序号}` 标注 |
| 节点数量 | 每层 3~9 个节点（BPMN 经验规则） |

### 步骤详情的必填字段

每个步骤节点必须包含：

| 字段 | 要求 | 示例 |
|------|------|------|
| 一句话目的 | 说明本步骤的作用 | "驱动精灵沿路线持续移动并检测终点" |
| 关键步骤 | 编号列表，每个动作一行，动词开头 | "1. 启动行军移动定时器" |
| 来源 | 关联 PRD 章节 | "[PRD 3.2.2 精灵出场](prd-link)" |
| 前置 | 从哪个步骤进入 | "← FL-XX-YY（条件）" |
| 后继 | 跳转到哪个步骤 | "→ FL-XX-ZZ" |
| 异常分支 | 条件 → 处理 → 转至 | "进度 ≥ 1.0 → 停止 → FL-XX-ZZ" |

可选字段（按需）：

| 字段 | 适用场景 |
|------|---------|
| ⏱ 时间 | 步骤有耗时/间隔/超时约束 |
| 📱 空间 | 步骤涉及视口/场景/布局变化 |
| 用户感知 | 步骤对用户可见的变化 |

## 3. 阶段三：穷举与补充

**目标**：对阶段二产出的流程树进行穷举补充，确保不遗漏任何步骤、交互、异常和数据操作。

### 步骤 3.1：交互行为穷举

对流程树中的每个步骤，逐一检查以下交互类型：

| 交互类型 | 检查项 |
|---------|--------|
| 点击 | 玩家点击了哪些可交互元素？ |
| 弹窗/面板 | 弹出了什么？如何关闭？ |
| 确认弹窗 | 确认/取消两条路径是否都有？ |
| 页面跳转 | 从哪跳到哪？触发条件？ |
| 选择/切换 | 选择后数据如何更新？ |
| 表单输入 | 输入校验规则？失败提示？ |
| 拖拽/滚动 | 拖拽后状态如何变化？ |

**弹窗闭环检查**：每个弹窗/面板必须满足：
1. 有明确的**弹出条件**
2. 有至少一种**关闭方式**（按钮/面板外/确认/自动）
3. 面板打开期间**背景流程**是否受影响必须说明

### 步骤 3.2：异常路径穷举

对每个正常步骤，系统性检查异常维度：

| 异常维度 | 检查问题 |
|---------|---------|
| 用户取消 | 从这个步骤能否取消？取消后如何清理？ |
| 校验失败 | 输入不合法时提示什么？停留在哪？ |
| 超时 | 是否有超时场景？超时后如何处理？ |
| 状态冲突 | 前置条件不满足时怎么办？ |
| 资源不足 | 资源不够时如何提示和处理？ |

**异常汇总表**（文档末尾必须有）：

```
| 触发步骤 | 异常条件 | 处理方式 | 恢复步骤 |
```

### 步骤 3.3：实体生命周期建模

**这是 v2 方法论的核心新增**。为每个关键实体定义完整的状态转换：

#### 实体状态机格式

```
实体名称
├── 状态列表
│   ├── S1: 空闲（初始状态）
│   ├── S2: 活跃
│   ├── S3: 完成
│   └── S4: 已销毁
├── 状态转换表
│   ├── S1 → S2: 触发条件 → 发生步骤（FL-XX-YY）
│   ├── S2 → S3: 触发条件 → 发生步骤（FL-XX-ZZ）
│   ├── S2 → S1: 触发条件（异常取消） → 发生步骤（FL-XX-AA）
│   └── S3 → S4: 触发条件 → 发生步骤（FL-XX-BB）
└── 关键字段
    ├── 字段A: 类型 | 创建时赋值 | 更新步骤: FL-XX-YY
    └── 字段B: 类型 | 创建时赋值 | 更新步骤: FL-XX-ZZ
```

#### 状态转换完整性检查（ESA 方法）

对每个实体，执行 **Events × States 笛卡尔积**检查：

```
对于实体 E:
  对于每个事件 Ev:
    对于每个状态 St:
      Ev 在 St 下是否有效？
      如果有效 → 转到哪个状态？在哪个流程步骤中定义？
      如果无效 → 是否需要防护（Guard）？
```

**关键要求**：
- 每个实体的**创建**步骤必须明确
- 每个实体的**销毁**步骤必须明确
- **所有终止路径**（正常完成、取消、异常）都必须清理实体

### 步骤 3.4：资源对称性检查

对每种资源操作，确认 lock/unlock 对称：

| 操作类型 | 锁定/扣减步骤 | 解锁/退还步骤（正常） | 解锁/退还步骤（取消） | 解锁/退还步骤（异常） |
|---------|-------------|-------------------|-------------------|-------------------|
| 资源 A 锁定 | FL-XX-01 | FL-XX-05 | FL-XX-CANCEL | — |
| 资源 B 扣减 | FL-XX-02 | — | FL-XX-CANCEL 退还 | — |

**检查规则**：
- 锁定的资源必须在**所有终止路径**上都有解锁
- 扣减的资源在取消时是否**退还**必须明确说明
- 兵力/英雄等"可损耗"资源，解锁时恢复的是**原始量**还是**剩余量**必须说明

## 4. 阶段四：一致性验证

**目标**：系统性验证流程树的完整性、一致性和可开发性。

### 步骤 4.1：需求覆盖验证（RTM）

更新阶段一创建的追溯矩阵，确保每个 PRD 需求都有对应的流程节点：

```
验证条件：
✓ 每个 PRD 需求 ID 至少有一个对应的流程节点
✓ 每个流程节点都有 PRD 来源
✓ PRD 中的边界条件已体现为异常分支
```

### 步骤 4.2：交互闭环验证

| 检查项 | 验证方法 |
|--------|---------|
| 弹窗闭环 | 每个弹窗都有至少一种关闭方式 |
| 确认弹窗 | 每个确认弹窗都有确认和取消两条路径 |
| 页面跳转 | 每个跳转都有明确的触发条件和目标 |
| 无死胡同 | 每个分支都有明确的出口 |

### 步骤 4.3：数据生命周期验证

| 检查项 | 验证方法 |
|--------|---------|
| 实体创建 | 每种实体都有明确的创建步骤 |
| 实体销毁 | 每种实体都有明确的销毁步骤 |
| 全路径清理 | 所有终止路径（正常/取消/异常）都清理了实体 |
| 资源对称 | 锁定/解锁、扣减/退还配对完整 |

### 步骤 4.4：跨树引用一致性

| 检查项 | 验证方法 |
|--------|---------|
| 前向引用 | A 树说"→ FL-XX"，B 树确实是 FL-XX 的定义 |
| 后向引用 | B 树的"前置"与 A 树的"后继"匹配 |
| 状态一致 | A 树设置的实体状态与 B 树期望的前置状态一致 |
| 无悬空引用 | 所有引用的 FL 编号都有对应定义 |
| 无矛盾路径 | A 树说"直接结束"，主流程说"应经过 B 树" — 禁止 |

### 步骤 4.5：步骤详情质量检查

| 检查项 | 要求 |
|--------|------|
| 每个步骤有目的 | 一句话说明作用 |
| 关键步骤格式 | 编号列表，每行一个动作，动词开头 |
| 不含 UI 细节 | 不描述颜色、像素、布局 |
| 不含不确定数值 | 用"参考 PRD"替代不确定的公式和数值 |
| 前置/后继明确 | 每个步骤都知道从哪来、到哪去 |

## 5. 文档规范

### 5.1 文档产出清单

一个完整的功能流程文档应包含以下产出物：

```
{功能名称}-结构化流程/
├── L1-{功能名称}-主流程.md          ← L1 主流程
├── L2-{子流程名称}.md               ← L2 子流程（每个 Phase 一个）
├── L3-{详细步骤名称}.md             ← L3 详细步骤（按需）
├── 实体生命周期.md                  ← 实体状态机（新增）
└── 需求追溯矩阵.md                  ← RTM（新增）
```

或者，对于较小的功能，可以合并为单个完整文档：

```
{功能名称}-结构化流程.md             ← 包含以上所有内容
```

### 5.2 实体生命周期文档格式

```markdown
## 实体：{实体名称}

> 定义：{一句话定义}
> PRD 来源：[PRD 章节](link)

### 状态列表

| 状态 | 说明 | 控制变量值 |
|------|------|-----------|
| 初始态 | ... | status = idle |
| 活跃态 | ... | status = active |
| 完成态 | ... | status = done |

### 状态转换表

| 当前状态 | 事件/条件 | 目标状态 | 触发步骤 | 伴随操作 |
|---------|----------|---------|---------|---------|
| 初始态 | 玩家确认 | 活跃态 | FL-XX-01 | 创建实例，锁定资源 |
| 活跃态 | 流程完成 | 完成态 | FL-XX-05 | 释放资源，更新记录 |
| 活跃态 | 玩家取消 | 初始态 | FL-XX-CANCEL | 退还资源，销毁实例 |

### 关键字段

| 字段 | 类型 | 初始值 | 更新步骤 |
|------|------|--------|---------|
| id | string | 自动生成 | 创建时 |
| status | enum | idle | FL-XX-01, FL-XX-05 |
```

### 5.3 需求追溯矩阵格式

```markdown
## 需求追溯矩阵

| PRD ID | 需求描述 | 流程节点 | 覆盖状态 |
|--------|---------|---------|---------|
| REQ-XX-01 | ... | FL-XX-01, FL-XX-02 | ✅ 已覆盖 |
| REQ-XX-02 | ... | FL-XX-03 | ✅ 已覆盖 |
| REQ-XX-03 | ... | — | ❌ 未覆盖 |
```

### 5.4 树形图绘制规范

```
根节点名称（FL-XX 流程名）
├── 01 步骤名称
│   ├── 子操作 A
│   ├── 子操作 B
│   └── → 子树引用（FL-XX-YY）
├── 02 步骤名称
│   └── ...
├── 异常: 条件 → 处理 → 转至
└── 异常: 条件 → 处理 → 转至
```

分支表示：
- 条件分支用 `→` 连接
- 异常分支以 `异常:` 开头，放在最后
- 子树引用用 `→ FL-{编号}` 或 `→ Tree {序号}` 标注
- 并行行为用 `（与 XX 并行）` 标注

## 6. Prompt 策略

### 6.1 四阶段 Prompt 模板

```
=== 阶段一：需求分析 ===
请阅读以下 PRD 片段，输出：
1. 功能边界（起点和终点）
2. 主流程骨干（3-7 个 Phase，每个一句话）
3. 关键实体列表（名称、定义、初始状态）
4. 需求 ID 列表（从 PRD 中提取或编号）

=== 阶段二：分层分解 ===
基于阶段一输出，请：
1. 生成 L1 主流程（ASCII 树 + 阶段详情）
2. 对每个 Phase 生成 L2 子流程（ASCII 树 + 步骤详情）
3. 对需要进一步展开的步骤生成 L3

规则提醒：
- 每棵树深度 ≤ 5 级
- 每个步骤有：目的、关键步骤、前置/后继、异常分支
- 关键步骤：编号列表，每行一个动作，动词开头
- 不写 UI 细节，不写不确定数值

=== 阶段三：穷举补充 ===
对阶段二产出的流程树，请逐一检查：
```
