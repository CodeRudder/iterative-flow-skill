## 一、问题：当前方法论遗漏什么

当前四阶段方法论（需求分析→分层分解→穷举补充→一致性验证）是**单视角自审**模式 — 同一个 AI 生成、审查、验证。这存在系统性盲区：

### 1.1 单视角自审的固有缺陷

| 缺陷 | 表现 | 根因 |
|------|------|------|
| **生成偏见** | 流程树偏向用户交互，忽略系统自动步骤 | 人类自然语言描述的用户视角偏重 |
| **验证盲区** | 自审容易跳过自己生成时已经忽略的点 | "没想到的就是没想到" |
| **隐性假设** | 实体状态机假设了某些隐含状态转换 | PRD 不写"理所当然"的东西 |
| **轴间脱节** | 流程树写了步骤但没有标注实体状态变更 | Flow 轴和 Data 轴独立生成后缺乏交叉验证 |
| **规则遗漏** | 条件路由的分支不完整，缺少默认兜底 | Rule 轴没有被独立视角审查 |

### 1.2 实际案例中的遗漏

回顾已生成的案例文档，以下是**自审未能发现但多视角审查可以发现的**问题类型：

**ERP 案例**：
- Flow 树写了"审批通过后检查预算"，但 Data 视角会追问："预算占用实体在审批中的状态是什么？审批进行中预算是否已被'预占用'？"
- 规则表定义了金额阈值路由，但 Rule 视角会追问："如果金额为 0（免费申请）怎么路由？规则表有兜底吗？"

**电商案例**：
- Flow 树写了"支付超时30分钟取消"，但 Data 视角会追问："超时取消时，库存锁定实体从'已锁定'转到什么状态？这个转换在状态机里有吗？"
- 异步回调有幂等检查，但 Rule 视角会追问："如果回调报文签名验证失败，是否重试？重试几次？"

**游戏案例**：
- Flow 树写了"行动力扣除"，但 Data 视角会追问："副本失败时行动力是否退还？退还在状态机的哪条转换路径？"

### 1.3 根因分析

```
当前流程（线性单视角）：

PRD → [单一AI视角] → Flow + Data + Role + Rule → [同一AI自审] → 修正

问题：生成和审查用的是同一个"思维模型"，看不到自己的盲区。
```

**核心洞察**：遗漏不是随机的 — 它们集中在**轴间交叉点**。Flow 不只看 Flow，Data 不只看 Data，而是 Flow×Data、Flow×Rule、Data×Role 的交叉区域最容易出现间隙。

## 二、解决方案：四轴模拟交叉审查

### 2.1 核心思想

借鉴 MARE 框架（arXiv:2405.03256）的 Checker Agent 和 MetaGPT（ICLR 2024）的 SOP 管道，设计一个**模拟多视角交叉审查**机制：

```
改进后流程（交叉视角）：

Phase X:
  Step 1: 并行生成四轴输出
    Flow Agent  → L2 子流程树（Phase X 的步骤和分支）
    Data Agent  → 实体状态变更（Phase X 涉及的实体操作）
    Role Agent  → 角色分配（Phase X 的步骤角色标注）
    Rule Agent  → 规则引用（Phase X 的条件路由和校验）

  Step 2: 交叉审查Agent模拟各轴视角互相提问
    以 Flow 视角审查 Data: "Data 说订单变为'已支付'，但 Flow 哪个步骤触发？"
    以 Data 视角审查 Flow: "Flow 步骤3扣减库存，但库存实体的状态转换表里有这一行吗？"
    以 Rule 视角审查 Flow: "Flow 在步骤2有条件分支，但引用了哪条规则？"
    以 Role 视角审查 Flow: "Flow 步骤4没标注角色，谁执行？"

  Step 3: 分类处理
    可从 PRD 推断 → 自动解决 + 记录决策
    标准模式有推荐 → 自动推荐 + 记录决策
    真正模糊/冲突 → 整理问题列表 → 用户决策

  Step 4: 输出
    更新后的 Phase X 四轴输出
    决策日志（已解决 + 待用户决策）
```

### 2.2 与现有方法论的关系

不是替代现有四阶段，而是**嵌入阶段二和阶段三之间**：

```
阶段一：需求分析 & 骨架提取     （不变）
    ↓
阶段二：分层流程分解             （不变，但按 Phase 逐个生成）
    ↓
★ 阶段 2.5：四轴交叉审查       （新增，每个 Phase 生成后立即执行）
    ↓
阶段三：穷举与补充               （不变，但审查发现的问题在此修复）
    ↓
阶段四：一致性验证               （不变，但质量应显著提高）
```

### 2.3 为什么选择"模拟多视角"而非"真正多Agent"

| 方案 | 优点 | 缺点 |
|------|------|------|
| 真正多Agent互问 | 审查最深入 | 实现极复杂，需要多轮Agent间通信，上下文管理困难，耗时 |
| 模拟多视角审查 | 实现简单，一个Prompt搞定，足够发现大部分问题 | 不如真正互问深入，但覆盖80%以上的间隙 |
| 单一审查 | 最简单 | 与当前自审无本质区别 |

**结论**：模拟多视角是**性价比最优**的选择。一个精心设计的Prompt可以让AI轮流扮演四个视角，每个视角有自己的审查清单。

## 三、交叉审查的具体问题清单

### 3.1 以 Flow 视角审查 Data

Flow Agent 关心的是**步骤完整性**，它看 Data 的状态机时会问：

| 审查项 | 具体问题 | 典型遗漏 |
|--------|---------|---------|
| 状态可达性 | Data 中每个状态，Flow 里都有步骤能到达吗？ | "已退款"状态存在但无步骤触发 |
| 转换触发 | Data 的每条状态转换，Flow 里都有对应步骤吗？ | 状态表有"超时→已取消"但流程无超时步骤 |
| 字段更新 | Data 的每个字段更新步骤，Flow 里存在吗？ | 字段表写了"更新步骤: FL-XX-05"但该步骤不存在 |
| 终态收敛 | Data 的每种终止路径，Flow 都有对应分支吗？ | 状态机有5条终止路径但流程只覆盖了3条 |

### 3.2 以 Data 视角审查 Flow

Data Agent 关心的是**数据一致性**，它看 Flow 的步骤树时会问：

| 审查项 | 具体问题 | 典型遗漏 |
|--------|---------|---------|
| 操作配对 | Flow 里的"锁定"/"占用"操作，有对应的"释放"步骤吗？ | 锁定了库存但取消路径未释放 |
| 状态标注 | Flow 的每步操作，对实体产生了什么状态变更？ | "审批通过"步骤没写实体变成什么状态 |
| 资源对称 | 所有终止路径上，每种资源操作都配对了吗？ | 正常路径释放了，超时路径没释放 |
| 字段来源 | Flow 引用的字段值，是从哪个步骤写入的？ | 步骤"读取审批金额"但金额是哪步写入的？ |

### 3.3 以 Rule 视角审查 Flow

Rule Agent 关心的是**条件完整性**，它看 Flow 的条件分支时会问：

| 审查项 | 具体问题 | 典型遗漏 |
|--------|---------|---------|
| 规则引用 | Flow 的每个条件分支，都引用了规则编号吗？ | 步骤有"依据金额路由"但没写规则编号 |
| 覆盖完整 | 规则表的所有条件组合，Flow 都有对应路径吗？ | 规则表有6条但流程只走了3条路径 |
| 兜底规则 | Flow 的条件分支最后有 else/默认路径吗？ | if-else 缺少 else 分支 |
| 规则矛盾 | 不同步骤引用的规则，有没有冲突？ | 步骤A说"≤5000主管审批"，步骤B说"≤10000主管审批" |

### 3.4 以 Role 视角审查 Flow

Role Agent 关心的是**职责清晰性**，它看 Flow 的步骤时会问：

| 审查项 | 具体问题 | 典型遗漏 |
|--------|---------|---------|
| 角色标注 | 每个步骤都标注了执行角色吗？ | "审批"步骤没说谁审批 |
| 角色连续性 | 前后步骤的角色切换合理吗？ | 步骤3是[员工]，步骤4突然变成[总经理]（缺少中间路由） |
| 权限互斥 | 同一角色是否既审批又提交？ | [员工]既创建申请又审批自己的申请 |
| 角色覆盖 | 每个角色至少负责一个步骤吗？ | 定义了5个角色但[总经理]没有对应步骤 |

### 3.5 跨轴交叉问题（最关键的间隙区域）

| 交叉区域 | 典型问题 |
|---------|---------|
| Flow × Data | "步骤扣了行动力，行动力实体在哪里初始分配？" |
| Flow × Rule | "步骤有条件分支，条件是什么？规则表有吗？" |
| Flow × Role | "步骤执行者是[系统]，但需要什么权限？谁配置的？" |
| Data × Rule | "规则说'预算不足驳回'，预算占用实体的状态转换表里有这条吗？" |
| Data × Role | "谁有权将实体状态从X改为Y？角色权限表里有吗？" |
| Rule × Role | "规则说'总经理审批'，但金额50万的审批，角色是部门经理还是总经理？" |

## 四、决策日志设计

### 4.1 日志结构

```markdown
## 决策日志 — Phase {X}: {阶段名称}

### 自动解决（{N} 项）

| # | 审查视角 | 发现的问题 | 解决方式 | 依据来源 | 影响范围 |
|---|---------|-----------|---------|---------|---------|
| A1 | Data→Flow | 步骤FL-03-02无实体状态变更标注 | 自动补充：采购申请单 → 审批中 | PRD §2.3 隐含 | FL-ERP-03-02 |
| A2 | Rule→Flow | 条件分支无规则引用 | 自动关联：R-01~R-03 | PRD §2.1 明确 | FL-ERP-03-01 |

### 推荐方案（{N} 项）

| # | 审查视角 | 发现的问题 | 推荐方案 | 推荐理由 | 状态 |
|---|---------|-----------|---------|---------|------|
| R1 | Data→Flow | 取消订单时库存锁定未释放 | 推荐在FL-05-06补充库存释放操作 | 资源对称性原则：锁定必须有释放 | ✓已采纳 |
| R2 | Rule→Flow | 金额=5000边界属于哪个规则 | 推荐使用 ≤5000 归R-01 | 行业惯例：左闭右开 | ?待确认 |

### 待用户决策（{N} 项）

| # | 审查视角 | 问题描述 | 上下文 | 各视角意见 | 用户回答 |
|---|---------|---------|--------|-----------|---------|
| Q1 | Flow×Data | 副本失败时行动力是否退还？ | PRD未明确说明。正常通关扣除，中途退出怎么处理？ | Flow: 应退还(用户未完成) / Data: 应退还(资源对称) / Rule: PRD未明确 | _____ |
| Q2 | Rule×Role | 部门经理缺席时审批如何流转？ | PRD §2.4定义了部门经理审批，但未说明替代方案 | Role: 可设代理人 / Rule: 可升级至总经理 | _____ |
```

### 4.2 决策类型判定逻辑

```
发现间隙
  │
  ├─ PRD 有明确文字描述？
  │    └─ 是 → 自动解决（类型A）
  │         记录：问题 + PRD出处 + 修复内容
  │
  ├─ PRD 未明确，但行业惯例/标准模式有推荐？
  │    └─ 是 → 推荐方案（类型R）
  │         记录：问题 + 推荐方案 + 推荐理由
  │         用户可一键采纳或修改
  │
  └─ PRD 未明确，且无公认标准？
       └─ 是 → 待用户决策（类型Q）
            记录：问题 + 上下文 + 各视角意见
            必须用户回答后才能继续
```

### 4.3 日志在整个流程中的作用

```
Phase A:
  生成 → 审查 → [决策日志A] → 用户确认Q类问题 → 修复 → Phase A完成
Phase B:
  生成 → 审查 → [决策日志B] → 用户确认Q类问题 → 修复 → Phase B完成
  ...
全部Phase完成:
  合并所有决策日志 → [全局决策总览] → 用户Review → 最终输出
```

## 五、交叉审查 Prompt 模板

### 5.1 主审查 Prompt

```
你是一个四轴交叉审查员，负责审查当前 Phase 的流程树完整性。

你的审查对象是以下四个轴的输出：
- Flow轴：{Phase_X_L2_流程树}
- Data轴：{Phase_X_实体状态变更}
- Role轴：{Phase_X_角色分配}
- Rule轴：{Phase_X_规则引用}

请按以下步骤执行交叉审查：

### 第一步：以 Flow 视角审查 Data
检查 Data 中的每个状态是否在 Flow 中有步骤可达，每条转换是否有步骤触发。
格式：[Flow→Data] 问题描述 | 严重程度(高/中/低) | 建议修复方式

### 第二步：以 Data 视角审查 Flow
检查 Flow 中的每步操作是否有对应的数据变更，资源操作是否配对。
格式：[Data→Flow] 问题描述 | 严重程度(高/中/低) | 建议修复方式

### 第三步：以 Rule 视角审查 Flow
检查 Flow 中的条件分支是否有规则引用，规则覆盖是否完整。
格式：[Rule→Flow] 问题描述 | 严重程度(高/中/低) | 建议修复方式

### 第四步：以 Role 视角审查 Flow
检查 Flow 中的步骤角色标注是否完整，职责是否清晰。
格式：[Role→Flow] 问题描述 | 严重程度(高/中/低) | 建议修复方式

### 第五步：跨轴交叉检查
检查 Flow×Data、Flow×Rule、Data×Rule 等交叉区域的间隙。
格式：[Flow×Data] 问题描述 | 严重程度(高/中/低) | 建议修复方式

### 第六步：分类
将所有发现的问题分为三类：
- **自动解决**：PRD 中有明确答案的，直接给出修复方案和 PRD 出处
- **推荐方案**：PRD 未明确但有行业标准/最佳实践的，给出推荐和理由
- **待用户决策**：PRD 未明确且无公认标准的，给出上下文和各视角意见

### 第七步：输出决策日志
按决策日志格式输出三类问题的完整列表。
```

### 5.2 用户决策 Prompt

```
以下是 Phase {X} 交叉审查发现的 {N} 个需要你决策的问题：

{Q类问题列表}

请逐个回答。你也可以选择：
- "采纳推荐" — 如果有推荐方案
- "跳过" — 如果你想后续再决定
- "自定义" — 如果你有不同的想法
```

## 六、整合进 iterative-flow 的完整流程

### 6.1 修改后的四阶段 + 审查

```
阶段一：需求分析 & 骨架提取
  → 输出：Phase列表、实体列表、角色列表、规则表（草稿）
  → 不做交叉审查（骨架阶段信息不足）

阶段二：分层流程分解（改为按Phase迭代）
  for each Phase:
    2.1 并行生成四轴输出
    2.2 执行交叉审查 ★新增
    2.3 整理决策日志
    2.4 如有待用户决策 → 暂停等待用户
    2.5 修复发现的问题
    2.6 确认本Phase完成 → 进入下一个Phase

阶段三：穷举与补充
  → 基于审查已修复的版本进行穷举
  → 审查已消除大部分轴间间隙，此阶段聚焦异常路径穷举

阶段四：一致性验证
  → 增加一项：审查决策日志中的"推荐方案"是否已全部处理
  → 其余验证项不变
```

### 6.2 输出文档变化

在原有文档结构基础上增加一个文件：

```
{功能名称}-结构化流程/
|-- ...
|-- 交叉审查日志.md              ← ★新增：所有Phase的审查决策记录
|-- 全局决策总览.md              ← ★新增：用户确认的决策汇总
```

### 6.3 对 SKILL.md 的影响

SKILL.md 需要增加一个步骤：在 L2 子流程生成后、穷举补充前，执行交叉审查。

## 七、预期效果

### 7.1 预期改善的问题类型

| 问题类型 | 当前发现率 | 预期发现率 | 改善机制 |
|---------|-----------|-----------|---------|
| 状态机与流程步骤脱节 | ~60% | ~95% | Data→Flow 交叉审查 |
| 资源操作不配对 | ~70% | ~95% | Data→Flow 资源对称检查 |
| 条件分支缺兜底 | ~50% | ~90% | Rule→Flow 覆盖完整性 |
| 步骤无角色标注 | ~80% | ~98% | Role→Flow 标注检查 |
| 跨轴间隙 | ~40% | ~85% | 跨轴交叉检查 |
| PRD 隐含假设 | ~30% | ~70% | 推荐方案触发用户思考 |

### 7.2 与 Grill Me 的本质区别

| 维度 | Grill Me | 四轴交叉审查 |
|------|---------|-------------|
| **追问者** | AI → 用户（单向） | 四轴互相（多向） |
| **追问对象** | 设计者本人 | 生成物（流程树/状态机） |
| **用户参与** | 全程参与 | 仅在无法自动解决时参与 |
| **问题来源** | AI 的好奇心 | 结构化的审查清单 |
| **可复现性** | 依赖AI当下的判断 | 审查清单固定，每次都检查 |
| **决策追溯** | 无结构化记录 | 决策日志有完整记录 |

## 九、结论

四轴模拟交叉审查可以有效补全 Flow Tree 的间隙，原因：

1. **结构性**：审查清单是固定的、穷举的，不依赖 AI 的"灵感"
2. **对抗性**：每个轴从自己的利益出发挑其他轴的毛病，比自审严格
3. **渐进性**：按 Phase 分批审查，问题发现早，修复成本低
4. **可追溯**：决策日志记录每个判断的来源和理由
5. **低侵入**：用户只在真正需要决策时才参与，不浪费注意力
