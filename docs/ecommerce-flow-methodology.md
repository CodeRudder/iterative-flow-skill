# 电商场景结构化流程树生成方法论

> 基于通用结构化流程树方法论 v2 的电商领域适配版本
> 聚焦订单、支付、库存、物流等电商核心流程的系统化建模
> 版本：v1.0 | 日期：2026-05-06

---

## 0. 核心设计原则

### 0.1 电商建模三轴：Flow + Data + Integration

电商系统在通用"流程 + 数据"双轴基础上，新增**外部集成**维度。流程树描述系统"做了什么"，实体生命周期描述系统"操作什么"，集成边界描述系统"与谁交互"。

| 轴 | 描述什么 | 输出物 |
|----|---------|--------|
| Flow 轴 | 步骤、分支、交互、异常 | 流程步骤树 |
| Data 轴 | 实体、状态、字段、约束 | 实体生命周期表 |
| Integration 轴 | 外部系统调用、回调、边界 | 集成边界说明 |

### 0.2 四阶段生成流程

```
阶段一：需求分析 & 骨架提取
    |
阶段二：分层流程分解
    |
阶段三：穷举与补充
    |
阶段四：一致性验证
```

### 0.3 电商特有核心关注点

| 关注点 | 说明 | 基础方法论覆盖情况 |
|--------|------|-------------------|
| 外部系统边界 | 支付网关、物流 API 等只描述接口不描述内部 | 需扩展 |
| 异步回调流程 | 支付回调、物流状态推送等两阶段建模 | 需扩展 |
| 订单状态机 | 电商订单拥有复杂且严格的状态转换 | 实体生命周期可覆盖 |
| 库存对称性 | 锁定/扣减/释放必须对称 | 资源对称性可覆盖 |
| 价格一致性 | 下单价格与支付价格必须一致 | 需扩展 |
| 幂等性 | 支付回调等必须幂等处理 | 需扩展 |

## 1. 阶段一：需求分析 & 骨架提取

**目标**：从 PRD 中提取电商流程骨架、关键实体和外部系统集成点。

### 步骤 1.1：PRD 结构化阅读

从 PRD 中提取以下信息：

| 提取项 | 说明 | 电商示例 |
|--------|------|---------|
| 功能边界 | 本流程的起点和终点 | "用户提交订单 -> 订单完成/关闭" |
| 用户角色 | 参与流程的角色 | 买家、卖家、平台运营、系统 |
| 主要场景 | 正常路径（Happy Path） | 下单 -> 支付 -> 发货 -> 收货 -> 完成 |
| 关键约束 | 时间、库存、金额约束 | "订单 30 分钟未支付自动取消" |
| 外部依赖 | 依赖的外部系统 | 支付宝/微信支付、顺丰/中通物流 API |
| 业务规则 | 促销、优惠、限购等规则 | "满减优惠不与单品折扣叠加" |

### 步骤 1.2：主流程骨干提取

以电商下单主流程为例，提取 3~7 个阶段：

```
主流程骨干（电商下单示例）
Phase A: 购物车确认    -- 用户确认购物车商品，选择收货地址
Phase B: 订单生成      -- 系统创建订单，锁定库存，计算价格
Phase C: 支付处理      -- 用户完成支付，系统确认收款
Phase D: 履约发货      -- 卖家发货，物流配送
Phase E: 确认收货      -- 买家确认收货，订单完成
```

**规则**（与基础方法论一致）：
- 每个阶段用一句话描述其核心目的
- 阶段按时间顺序排列
- 阶段数量控制在 3~7 个
- 每个阶段对应一棵 L2 子流程树

### 步骤 1.3：关键实体识别

电商场景下的核心实体及其生命周期：

| 实体 | 定义 | 初始状态 | 来源 |
|------|------|---------|------|
| 购物车 (Cart) | 用户待结算商品的容器 | 空闲 | PRD 购物车章节 |
| 订单 (Order) | 一次交易的核心凭证 | 不存在 | PRD 订单系统章节 |
| 支付单 (Payment) | 一次支付的记录 | 未创建 | PRD 支付系统章节 |
| 库存锁定 (InventoryLock) | 对商品库存的临时占用 | 未锁定 | PRD 库存系统章节 |
| 物流单 (Shipment) | 商品配送的跟踪记录 | 未创建 | PRD 物流系统章节 |
| 退款单 (Refund) | 退款申请及处理记录 | 未创建 | PRD 售后系统章节 |

### 步骤 1.4：外部系统集成点识别

**电商方法论新增步骤**。明确识别所有外部系统交互点：

| 外部系统 | 交互类型 | 我方职责 | 对方职责（不建模） |
|---------|---------|---------|-----------------|
| 支付宝/微信支付 | 支付发起 + 回调接收 | 创建支付请求、处理回调通知 | 验证密码、扣款、发送回调 |
| 物流服务商 | 物流下单 + 状态查询 | 创建物流订单、接收状态推送 | 取件、配送、状态更新 |
| 库存系统（内部） | 库存锁定/扣减/释放 | 发起库存操作请求 | 库存数量管理 |
| 消息服务（内部） | 短信/推送通知 | 发送通知请求 | 消息投递 |

**集成边界规则**：
- 只描述本系统发起的调用和接收的回调
- 不描述外部系统的内部处理逻辑
- 使用"调用 X -> 等待回调"模式描述异步交互
- 外部系统的数据结构只记录本系统需要关注的字段

### 步骤 1.5：建立需求可追溯性矩阵（草稿）

与基础方法论一致，为每个 PRD 需求分配唯一 ID：

| PRD ID | 需求描述 | 对应阶段 | 流程节点（待填充） |
|--------|---------|---------|------------------|
| REQ-ORD-01 | 用户可从购物车提交订单 | Phase B | -- |
| REQ-ORD-02 | 订单创建时锁定库存 | Phase B | -- |
| REQ-PAY-01 | 支持支付宝和微信支付 | Phase C | -- |
| REQ-PAY-02 | 支付超时 30 分钟自动取消 | Phase C | -- |
| REQ-SHIP-01 | 卖家发货后生成物流单号 | Phase D | -- |

## 2. 阶段二：分层流程分解

**目标**：将骨干逐层分解为可执行的流程步骤树，重点处理电商特有的异步流程和状态机。

### 步骤 2.1：生成 L1 主流程

以电商下单主流程为例：

```
FL-ORD-01 电商下单主流程
|-- Phase A: 购物车确认 -> FL-ORD-02
|-- Phase B: 订单生成 -> FL-ORD-03
|-- Phase C: 支付处理 -> FL-ORD-04
|   |-- 异常: 支付超时 -> 取消订单 -> FL-ORD-CANCEL
|   |-- 异常: 支付失败 -> 重新支付/取消 -> FL-ORD-CANCEL
|-- Phase D: 履约发货 -> FL-ORD-05
|-- Phase E: 确认收货 -> FL-ORD-06
|   |-- 异常: 超时自动确认 -> Phase E 完成
|-- FL-ORD-CANCEL: 订单取消（异常路径）
```

**L1 主流程组成**（与基础方法论一致）：元信息、流程概览、阶段详情。

### 步骤 2.2：生成 L2 子流程

以 Phase C 支付处理为例：

```
FL-ORD-04 支付处理子流程
|-- 01 展示支付方式选择页
|   |-- 加载订单金额
|   |-- 展示可用支付方式（支付宝、微信支付）
|   |-- 展示支付剩余时间倒计时
|-- 02 用户选择支付方式
|   |-- 用户点击支付方式
|   |-- 确认支付金额
|-- 03 创建支付单
|   |-- 生成支付单号（幂等键）
|   |-- 记录支付方式、金额、关联订单
|   |-- 调用支付渠道创建预付单
|-- 04 跳转支付页面
|   |-- 获取支付渠道跳转 URL
|   |-- 重定向用户到支付页面
|   |-- 启动支付超时倒计时（30 分钟）
|-- 05 等待支付回调
|   |-- 接收支付渠道异步通知
|   |-- 验证回调签名
|   |-- 幂等检查（支付单状态是否为 pending）
|   |-- 更新支付单状态和订单状态
|-- 异常: 支付超时 -> 查询支付结果 -> 关闭支付单 -> 释放库存 -> 取消订单
|-- 异常: 用户主动取消 -> 关闭支付单 -> 释放库存 -> 取消订单
|-- 异常: 回调签名校验失败 -> 记录日志 -> 忽略回调（等待重试或超时）
```

### 步骤 2.3：异步流程建模规则

**电商特有规则**。电商系统大量使用异步交互，必须按以下模式建模：

#### 两阶段异步模式

每个涉及外部系统的异步操作，必须建模为两个独立步骤：

```
异步操作建模规范：

步骤 A（发起请求）:
  1. 构造请求参数
  2. 调用外部系统接口
  3. 记录请求 ID 和请求时间
  4. 设置超时定时器
  5. 本系统状态变更为"已发起（pending）"

步骤 B（接收回调）:
  1. 接收回调通知
  2. 验证回调来源和签名
  3. 幂等性检查（查询本系统状态）
  4. 若已是终态则跳过处理并返回成功
  5. 执行业务逻辑，更新本系统状态
  6. 清除超时定时器
  7. 返回确认响应给外部系统
```

**典型电商异步场景**：

| 异步场景 | 发起阶段 | 回调阶段 | 超时处理 |
|---------|---------|---------|---------|
| 支付 | 创建预付单并跳转 | 支付渠道异步通知 | 30 分钟后主动查询并关闭 |
| 物流下单 | 调用物流 API 创建运单 | 物流接单回调通知 | 重试 3 次后转人工处理 |
| 物流状态追踪 | 订阅物流状态推送 | 物流状态变更回调 | 定时轮询补偿 |
| 退款 | 调用退款 API | 退款渠道异步通知 | 24 小时后主动查询结果 |

#### 幂等性规则

电商异步回调必须处理幂等：

```
幂等处理规则：
1. 每个外部请求携带唯一幂等键（如支付单号）
2. 接收回调时先查询本系统实体状态
3. 如果实体状态已经是终态，直接返回成功确认，不重复处理
4. 如果实体状态为中间态，执行状态转换逻辑
5. 所有回调处理必须记录日志，包含幂等键和处理结果
```

### 步骤 2.4：子树拆分规则

与基础方法论一致，补充电商特有拆分时机：

| 规则 | 说明 |
|------|------|
| 最大深度 | 每棵树不超过 5 级 |
| 异步拆分 | 异步回调处理独立拆为子树 |
| 状态机拆分 | 实体状态转换逻辑独立拆为子树 |
| 补偿拆分 | 超时/失败补偿流程独立拆为子树 |
| 节点数量 | 每层 3~9 个节点 |

### 步骤详情的必填字段

与基础方法论一致，电商场景增加以下可选字段：

| 字段 | 适用场景 | 电商示例 |
|------|---------|---------|
| 幂等键 | 涉及外部调用的步骤 | "支付单号作为幂等键" |
| 超时时间 | 有超时约束的步骤 | "支付超时 30 分钟" |
| 补偿操作 | 异步操作失败后的处理 | "超时后主动查询支付结果" |
| 价格校验 | 涉及金额变更的步骤 | "校验支付金额 = 订单应付金额" |
| 库存操作 | 涉及库存变更的步骤 | "锁定 SKU-A 库存 2 件" |

## 3. 阶段三：穷举与补充

**目标**：对阶段二产出的流程树进行穷举补充，重点覆盖电商特有的异常路径、状态机和一致性约束。

### 步骤 3.1：交互行为穷举

对流程树中的每个步骤，逐一检查以下交互类型：

| 交互类型 | 检查项 | 电商示例 |
|---------|--------|---------|
| 点击 | 用户点击了哪些可交互元素？ | "点击支付按钮"、"点击收货地址选项" |
| 弹窗/面板 | 弹出了什么？如何关闭？ | "优惠详情弹窗，点击关闭或外部区域关闭" |
| 确认弹窗 | 确认/取消两条路径是否都有？ | "取消订单确认弹窗：确认取消 / 继续支付" |
| 页面跳转 | 从哪跳到哪？触发条件？ | "选择支付后跳转支付宝收银台" |
| 选择/切换 | 选择后数据如何更新？ | "切换收货地址后重新计算运费" |
| 表单输入 | 输入校验规则？失败提示？ | "手机号格式校验，失败提示请输入正确手机号" |

**弹窗闭环检查**（与基础方法论一致）：每个弹窗必须有明确的弹出条件、关闭方式和背景流程说明。

### 步骤 3.2：异常路径穷举

对每个正常步骤，系统性检查异常维度：

| 异常维度 | 检查问题 | 电商示例 |
|---------|---------|---------|
| 用户取消 | 从这个步骤能否取消？取消后如何清理？ | "支付页取消 -> 关闭支付单 -> 释放库存" |
| 校验失败 | 输入不合法时提示什么？停留在哪？ | "地址信息不完整 -> 提示补充 -> 停留在地址页" |
| 超时 | 是否有超时场景？超时后如何处理？ | "支付超时 30 分钟 -> 主动查询结果 -> 取消订单" |
| 状态冲突 | 前置条件不满足时怎么办？ | "库存不足 -> 提示库存变化 -> 返回购物车" |
| 资源不足 | 资源不够时如何提示和处理？ | "库存不足 -> 提示商品已售罄 -> 建议类似商品" |
| 外部系统异常 | 外部服务不可用时如何处理？ | "支付网关超时 -> 提示稍后重试 -> 不自动取消" |

**异常汇总表**（文档末尾必须有）：

```
| 触发步骤 | 异常条件 | 处理方式 | 恢复步骤 |
```

### 步骤 3.3：订单状态机建模

**电商方法论核心步骤**。电商订单拥有复杂且严格的状态转换，必须作为一等实体建模。

#### 订单状态机定义

```
订单状态机（Order State Machine）

状态列表：
  S1: 待支付 (pending_payment)    -- 订单已创建，等待用户支付
  S2: 已支付 (paid)               -- 支付成功，等待卖家发货
  S3: 待发货 (pending_shipment)   -- 订单已分配给仓库，等待出库
  S4: 已发货 (shipped)            -- 卖家已发货，物流在途
  S5: 已签收 (delivered)          -- 买家已签收
  S6: 已完成 (completed)          -- 订单完成（确认收货或自动确认）
  S7: 已取消 (cancelled)          -- 订单已取消
  S8: 退款中 (refunding)          -- 退款申请已提交，等待处理
  S9: 已退款 (refunded)           -- 退款完成
  S10: 已关闭 (closed)            -- 订单终态（退款后关闭或超时关闭）

状态转换表：
  S1 -> S2:  支付成功回调          -- FL-ORD-04-05
  S1 -> S7:  用户取消/支付超时     -- FL-ORD-CANCEL
  S2 -> S3:  订单分配仓库          -- FL-ORD-05-01
  S3 -> S4:  卖家发货              -- FL-ORD-05-03
  S4 -> S5:  物流签收回调          -- FL-ORD-06-01
  S5 -> S6:  买家确认收货/超时     -- FL-ORD-06-02
  S2 -> S8:  买家申请退款          -- FL-REFUND-01
  S3 -> S8:  买家申请退款          -- FL-REFUND-01
  S4 -> S8:  买家申请退款          -- FL-REFUND-01
  S8 -> S9:  退款成功              -- FL-REFUND-04
  S8 -> S2:  退款被拒绝            -- FL-REFUND-03
  S9 -> S10: 退款完成关闭          -- FL-REFUND-05
  S7 -> S10: 取消完成关闭          -- FL-ORD-CANCEL
```

**状态转换完整性检查（ESA 方法）**：

对订单状态机执行 Events x States 笛卡尔积检查：

```
对于订单实体 Order:
  对于每个事件 Ev（支付成功、支付超时、用户取消、卖家发货、物流签收、确认收货、申请退款、退款完成、退款拒绝）:
    对于每个状态 St (S1~S10):
      Ev 在 St 下是否有效？
      如果有效 -> 转到哪个状态？在哪个流程步骤中定义？
      如果无效 -> 是否需要防护（Guard）？是否需要提示？
```

#### 关键规则：
- 每个实体的**创建**步骤必须明确（订单在 FL-ORD-03-01 创建）
- 每个实体的**终态**步骤必须明确（completed、closed 为终态）
- **所有终止路径**都必须到达终态，不可停留在中间态
- **禁止非法跳转**：如从 paid 直接跳到 cancelled 必须经过退款流程

### 步骤 3.4：其他核心实体生命周期

#### 购物车状态机

```
购物车（Cart）
  状态: idle -> active -> locked -> settled / abandoned
  idle -> active:   用户添加第一件商品
  active -> locked: 用户点击"去结算"，进入下单流程
  locked -> settled: 订单创建成功，清空购物车
  locked -> active: 下单失败/取消，解锁购物车
  active -> abandoned: 用户长时间未操作（可选，不删除数据）
```

#### 支付单状态机

```
支付单（Payment）
  状态: created -> pending -> success / failed / expired
  created -> pending:  调用支付渠道创建预付单成功
  pending -> success:  支付渠道回调通知支付成功
  pending -> failed:   支付渠道回调通知支付失败
  pending -> expired:  支付超时，主动查询确认后关闭
  created -> expired:  创建预付单失败，直接关闭
```

#### 库存锁定状态机

```
库存锁定（InventoryLock）
  状态: locked -> deducted / released
  (无锁定) -> locked:    订单创建时锁定库存
  locked -> deducted:    支付成功后扣减库存
  locked -> released:    订单取消/支付失败/超时后释放库存
```

#### 物流单状态机

```
物流单（Shipment）
  状态: not_created -> pending -> shipped -> delivered
  not_created -> pending: 卖家点击发货，调用物流 API
  pending -> shipped:     物流公司确认揽件
  shipped -> delivered:   物流签收
```

### 步骤 3.5：库存操作对称性检查

**电商方法论核心检查项**。库存操作必须严格对称：

```
库存对称性检查表：

| 操作类型         | 锁定/扣减步骤     | 正常释放             | 取消释放             | 异常释放             |
|----------------|-----------------|--------------------|--------------------|--------------------|
| 库存锁定         | FL-ORD-03-02    | FL-ORD-04-05 (扣减) | FL-ORD-CANCEL (释放) | 支付超时 (释放)      |
| 库存扣减         | FL-ORD-04-05    | -- (终态)           | FL-REFUND-04 (退还)  | --                  |
| 优惠券锁定       | FL-ORD-03-03    | FL-ORD-04-05 (核销) | FL-ORD-CANCEL (退回) | 支付超时 (退回)      |
```

**检查规则**：
- 锁定的库存必须在**所有终止路径**上都有解锁（扣减或释放）
- 每次库存操作必须记录操作日志（锁定量、操作人、关联订单）
- 库存锁定必须有超时机制，防止永久锁定
- 库存释放时恢复的是**原始锁定量**（不是剩余量）

### 步骤 3.6：价格一致性检查

**电商方法论新增检查项**。确保价格在整个流程中保持一致：

```
价格一致性检查规则：
1. 订单创建时快照商品价格 -> 记录到订单明细
2. 支付创建时校验 -> 支付金额 = 订单应付金额
3. 支付回调时校验 -> 回调金额 = 支付单金额
4. 退款时校验 -> 退款金额 <= 实付金额
5. 优惠分摊校验 -> 各商品优惠分摊之和 = 总优惠金额
```

## 4. 阶段四：一致性验证

**目标**：系统性验证流程树的完整性、一致性和可开发性，加入电商特有验证维度。

### 步骤 4.1：需求覆盖验证（RTM）

更新阶段一创建的追溯矩阵，确保每个 PRD 需求都有对应的流程节点：

```
验证条件：
- 每个 PRD 需求 ID 至少有一个对应的流程节点
- 每个流程节点都有 PRD 来源
- PRD 中的边界条件已体现为异常分支
- 电商特有需求（支付方式、超时策略、退款规则）全部覆盖
```

### 步骤 4.2：交互闭环验证

| 检查项 | 验证方法 |
|--------|---------|
| 弹窗闭环 | 每个弹窗都有至少一种关闭方式 |
| 确认弹窗 | 每个确认弹窗都有确认和取消两条路径 |
| 页面跳转 | 每个跳转都有明确的触发条件和目标 |
| 无死胡同 | 每个分支都有明确的出口 |
| 支付流程闭环 | 跳转到支付页面后必须有回跳或超时处理 |

### 步骤 4.3：数据生命周期验证

| 检查项 | 验证方法 |
|--------|---------|
| 实体创建 | 每种实体都有明确的创建步骤 |
| 实体终态 | 每种实体都能到达合法的终态 |
| 全路径清理 | 所有终止路径（正常/取消/异常）都清理了实体 |
| 资源对称 | 库存锁定/解锁、优惠券锁定/退回配对完整 |

### 步骤 4.4：电商特有验证

**以下为电商方法论新增验证项**，必须逐一检查：

#### 4.4.1 支付幂等性验证

```
检查项：
- 支付回调处理是否有幂等键
- 幂等键是否为支付单号（唯一）
- 重复回调是否被正确忽略（返回成功但不重复处理）
- 幂等处理是否有日志记录
```

#### 4.4.2 库存一致性验证

```
检查项：
- 锁定量 = 扣减量 + 释放量（数学恒等式）
- 所有取消路径是否都释放了库存
- 支付超时路径是否都释放了库存
- 退款路径是否退还了库存（已扣减的情况）
- 是否存在库存锁定后无法释放的死路径
```

#### 4.4.3 订单状态合法转换验证

```
检查项：
- 所有状态转换是否在状态机中有定义
- 是否存在非法跳转（如 paid 直接到 cancelled 没有经过退款）
- 终态（completed、closed）之后是否还有转换
- 并发场景下状态转换是否有防护（如同时收到回调和超时）
```

#### 4.4.4 价格一致性验证

```
检查项：
- 订单创建价格是否快照（防止商品价格变动影响已创建订单）
- 支付金额是否与订单应付金额一致
- 退款金额是否不超过实付金额
- 优惠金额分摊是否正确
```

### 步骤 4.5：跨树引用一致性

| 检查项 | 验证方法 |
|--------|---------|
| 前向引用 | A 树说"-> FL-ORD-04"，B 树确实是 FL-ORD-04 的定义 |
| 后向引用 | B 树的"前置"与 A 树的"后继"匹配 |
| 状态一致 | A 树设置的实体状态与 B 树期望的前置状态一致 |
| 无悬空引用 | 所有引用的 FL 编号都有对应定义 |
| 无矛盾路径 | A 树说"直接结束"，主流程说"应经过 B 树" -- 禁止 |
| 回调注册一致 | 发起异步请求的步骤说"等待回调"，回调处理步骤确实存在 |

### 步骤 4.6：步骤详情质量检查

| 检查项 | 要求 |
|--------|------|
| 每个步骤有目的 | 一句话说明作用 |
| 关键步骤格式 | 编号列表，每行一个动作，动词开头 |
| 不含 UI 细节 | 不描述颜色、像素、布局 |
| 不含不确定数值 | 用"参考 PRD"替代不确定的公式和数值 |
| 前置/后继明确 | 每个步骤都知道从哪来、到哪去 |
| 异步步骤成对 | 每个异步发起都有对应的回调处理步骤 |

### 6.2 电商场景 ToT 分支穷举

对电商流程中的决策点，使用 Tree of Thought 展开：

```
在步骤 FL-ORD-04-05（支付回调处理）中，有以下决策点：
  决策点 1: 支付结果
    |-- 支付成功 -> 更新支付单状态 -> 更新订单状态 -> 扣减库存 -> 进入发货流程
    |-- 支付失败 -> 更新支付单状态 -> 释放库存 -> 订单状态改为待支付（允许重试）
    |-- 支付处理中 -> 不更新状态 -> 等待下次回调或超时查询
  决策点 2: 幂等检查
    |-- 首次回调 -> 正常处理
    |-- 重复回调且已成功 -> 返回成功确认，不重复处理
    |-- 重复回调但状态不一致 -> 记录告警日志，以当前实际状态为准
```
